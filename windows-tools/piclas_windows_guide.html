piclas-win 0.9

Unofficial Windows port of PICLas 4.1.0 — complete setup, build, and distribution guide for Windows 11 with MSYS2 UCRT64

⚠ This is an unofficial Windows port. It is not affiliated with or endorsed by the PICLas development team. It is based on PICLas 4.1.0 (commit d63756ee) with Windows-specific patches applied on top. For the official Linux version, see github.com/piclas-framework/piclas.

📅 April 2026 🏴 Windows 11 Pro 🔧 MSYS2 UCRT64 / GCC 📄 Based on PICLas v4.1.0

Overview

PICLas is a Particle-in-Cell / Direct Simulation Monte Carlo (PIC-DSMC) plasma simulation code written in Fortran/C. It was originally developed for Linux/HPC environments. This guide documents all steps taken to build and run PICLas on Windows 11 using the MSYS2 UCRT64 environment, including source code patches, build system configuration, and runtime fixes.

Key strategy PICLas is compiled natively on Windows using GCC/GFortran provided by MSYS2 (not Cygwin, not WSL). The resulting piclas.exe is a real Windows executable that runs without MSYS2 at runtime, provided the required DLLs are bundled (see Distribution).

4. Source Code Changes

PICLas requires four modifications to build and run correctly on Windows. These changes are backward-compatible: on Linux/macOS the new C file compiles to a no-op stub and behaviour is unchanged.

4.1 Windows Wildcard Expansion — glob_windows.c NEW FILE

On Linux, the shell expands wildcards like plasma_wave_State_*.h5 before passing them to the program. Windows shells (CMD, PowerShell) do not — the literal string * reaches the program unchanged. This new file implements wildcard expansion using the Win32 API so that piclas2vtk parameter.ini State_*.h5 works from any Windows terminal.

File: piclas-master/src/globals/glob_windows.c 
// Win32 glob expansion — called from Fortran via ISO_C_BINDING
// On non-Windows: stub always returns 0 (shell already expands globs)
#ifdef _WIN32
#include <windows.h>
#include <string.h>
#include <stdlib.h>

static int cmp_str(const void *a, const void *b) {
    return strcmp(*(const char **)a, *(const char **)b);
}

int glob_expand_c(const char *pattern, char *results, int results_len) {
    WIN32_FIND_DATAA fd;
    HANDLE h;
    char *ptrs[4096];
    int count = 0;
    int pos = 0;

    h = FindFirstFileA(pattern, &fd);
    if (h == INVALID_HANDLE_VALUE) return 0;

    /* Determine directory prefix from pattern */
    char dir[MAX_PATH] = "";
    const char *slash = strrchr(pattern, '/');
    const char *bslash = strrchr(pattern, '\\');
    const char *last = slash > bslash ? slash : bslash;
    if (last) { int n = (int)(last - pattern) + 1; memcpy(dir, pattern, n); dir[n] = 0; }

    do {
        if (count >= 4096) break;
        char *entry = malloc(MAX_PATH);
        snprintf(entry, MAX_PATH, "%s%s", dir, fd.cFileName);
        ptrs[count++] = entry;
    } while (FindNextFileA(h, &fd));
    FindClose(h);

    qsort(ptrs, count, sizeof(char *), cmp_str);

    for (int i = 0; i < count; i++) {
        int len = strlen(ptrs[i]);
        if (pos + len + 1 >= results_len) { free(ptrs[i]); break; }
        memcpy(results + pos, ptrs[i], len + 1);
        pos += len + 1;
        free(ptrs[i]);
    }
    return count;
}
#else
int glob_expand_c(const char *pattern, char *results, int results_len) {
    (void)pattern; (void)results; (void)results_len;
    return 0;  /* Linux/macOS: shell handles glob expansion */
}
#endif

4.2 Fortran Command-Line Interface — commandlinearguments.f90 MODIFIED

Add an ISO_C_BINDING interface for glob_expand_c and call it for any argument containing * or ?.

File: piclas-master/src/globals/commandlinearguments.f90

Add at the top of the module, inside the relevant subroutine:

USE ISO_C_BINDING, ONLY: C_INT, C_CHAR, C_NULL_CHAR

INTERFACE
  FUNCTION glob_expand_c(pattern, results, results_len) BIND(C, NAME='glob_expand_c')
    USE ISO_C_BINDING, ONLY: C_INT, C_CHAR
    CHARACTER(KIND=C_CHAR), INTENT(IN)        :: pattern(*)
    CHARACTER(KIND=C_CHAR), INTENT(OUT)       :: results(*)
    INTEGER(KIND=C_INT),    VALUE, INTENT(IN) :: results_len
    INTEGER(KIND=C_INT)                       :: glob_expand_c
  END FUNCTION glob_expand_c
END INTERFACE

Then wrap each argument that contains * or ? through glob_expand_c before storing it.

5. Build Fixes

5.1 TEMP/TMP environment variables

On some Windows systems, the TEMP and TMP variables point to C:\Windows\Temp (a system directory). GCC cannot write temporary files there and aborts with "Cannot create temporary file in C:\WINDOWS\: Permission denied".

Set them to the user temp directory before building. Add these lines to any build script:

export TEMP='/c/Users/YOUR_USERNAME/AppData/Local/Temp'
export TMP=$TEMP

Or in a .bat file, use /tmp which MSYS2 maps to C:\msys64\tmp:

"%BASH%" -c "export TEMP=/tmp && export TMP=/tmp && cmake --build ..."

5.2 Missing Fortran module include directory

GFortran warns about C:/msys64/ucrt64/mod/static not existing. The directory is referenced by the HDF5 Fortran CMake configuration for static library modules. The warning is harmless but can be silenced by creating the empty directory: 

mkdir -p /c/msys64/ucrt64/mod/static

5.3 PETSc pkg-config wrapper FIX

MSYS2 installs PETSc with variant-specific .pc files (petsc-dso.pc, petsc-cso.pc, etc.) but not a plain petsc.pc. PICLas's FindPETSc.cmake uses pkg-config to search for the package name petsc and, when it fails to find it, falls back to downloading and compiling PETSc from source — which fails on Windows because the PETSc configure script uses Unix environment-variable syntax incompatible with cmd.exe.

The fix is to create a plain petsc.pc wrapper pointing to the dso variant (double precision, serial, optimized): 

# Create file: C:\msys64\ucrt64\lib\pkgconfig\petsc.pc
prefix=/c/msys64/ucrt64
exec_prefix=${prefix}
libdir=${exec_prefix}/lib/petsc/dso
includedir=${exec_prefix}/include/petsc

Name: PETSc
Description: Sequential double precision PETSc build
Version: 3.24.5
Requires.private: hwloc openblas
Cflags: -I${includedir}/dso -I${includedir}
Libs: -L${libdir} -lpetsc
Libs.private: -L${libdir} -lpetsc -lgfortran -lquadmath

Create this file from the MSYS2 shell:

cat > /c/msys64/ucrt64/lib/pkgconfig/petsc.pc << 'EOF'
prefix=/c/msys64/ucrt64
exec_prefix=${prefix}
libdir=${exec_prefix}/lib/petsc/dso
includedir=${exec_prefix}/include/petsc

Name: PETSc
Description: Sequential double precision PETSc build
Version: 3.24.5
Requires.private: hwloc openblas
Cflags: -I${includedir}/dso -I${includedir}
Libs: -L${libdir} -lpetsc
Libs.private: -L${libdir} -lpetsc -lgfortran -lquadmath
EOF

Verify it works:

pkg-config --modversion petsc
# Expected output: 3.24.5

When building with PETSc, also pass these cmake flags to ensure the system library is used rather than triggering a source build:

-DLIBS_BUILD_PETSC=OFF \
-DPETSC_LINK_LIBRARIES=C:/msys64/ucrt64/lib/petsc/dso/libpetsc.dll.a

And set the environment variable so FindPETSc.cmake finds the pkgconfig directory:

export PETSC_DIR=/c/msys64/ucrt64

5.4 Parallel build race condition (unit tests)

When Ninja builds in parallel, unit test source files can start compiling before the Fortran .mod files in libpiclasstaticF90 have been generated, causing "Fatal Error: Cannot open module file 'mod_globals.mod'".

Workaround: build with --target piclas to skip unit tests, or set PICLAS_UNITTESTS=OFF. The CMake dependency fix in unitTests/CMakeLists.txt (Section 4.3) also mitigates this.

6. MPI Build (MS-MPI)

PICLas can be built with MPI support on Windows using Microsoft MPI (MS-MPI v10.1) provided by MSYS2. This enables multi-process parallel simulations via mpiexec -n N piclas.exe. Note that MSYS2's HDF5 is built without parallel MPI support, so file I/O remains serial even in MPI runs — this is the standard behaviour for most desktop/workstation use cases.

6.1 Install MS-MPI Package

Install the MSYS2 MS-MPI wrapper package (provides mpicc, mpifort, import library libmsmpi.dll.a, and header mpi.h):

pacman -S --noconfirm mingw-w64-ucrt-x86_64-msmpi
MS-MPI runtime The MSYS2 package provides the import library for linking. For running MPI programs you also need the MS-MPI runtime DLL (msmpi.dll). This is provided by Microsoft MPI v10 and is typically already installed as part of the MSYS2 package or can be installed separately.

6.2 Source Code and CMake Fixes

Eight fixes are needed to get the MPI build working correctly on Windows. These are all already applied in the patched source tree.

Fix A — mpi_f08.o linker ordering (cmake/SetLibraries.cmake)

MS-MPI v10 provides only the legacy MODULE MPI interface. PICLas ships a compatibility wrapper cmake/mpi_f08.f90 that provides the modern typed MPI_F08 interface, compiled into mpi_f08.o. GNU ld scans archives left-to-right: the object file must appear before libmsmpi.dll.a so that its undefined references (mpi_wtick_ etc.) are known when the import library is scanned. Changed from LIST(APPEND…) to LIST(INSERT … 0 …): 

# mpi_f08.o must precede libmsmpi.dll.a in the link command
IF(EXISTS "${_mpi_f08_dir}/mpi_f08.o")
  LIST(INSERT linkedlibs 0 "${_mpi_f08_dir}/mpi_f08.o")
ENDIF()

Fix B — HDF5 parallel detection false positive (cmake/SetLibraries.cmake)

MSYS2 HDF5 is built without parallel MPI support (HDF5_PROVIDES_PARALLEL=OFF in hdf5-config.cmake). The original detection used grep mpio_f which matched the symbol h5fd_mpio_f_ (an MPIO file-driver constant that is always present), causing a false HDF5_IS_PARALLEL=TRUE. Changed to search for the specific function that only exists in parallel-capable builds: 

# Search specifically for h5pset_fapl_mpio_f to avoid false positives from h5fd_mpio_f
ELSEIF(WIN32)
  EXECUTE_PROCESS(COMMAND nm -g ${HDF5_Fortran_LIBRARY_hdf5_fortran}
                  COMMAND grep h5pset_fapl_mpio_f
                  OUTPUT_VARIABLE HDF5_USES_MPIF ...)

Fix C — USE_MPI_HDF5 compile flag + source guards (cmake/SetLibraries.cmake, 3 HDF5 source files)

Added a new preprocessor flag that is 1 only when both MPI is enabled AND HDF5 was built with parallel support. This guards the 5 places in the source that call H5PSET_FAPL_MPIO_F or H5PSET_DXPL_MPIO_F (which do not exist in the serial MSYS2 HDF5 library): 

# cmake/SetLibraries.cmake — after HDF5 detection:
IF(LIBS_USE_MPI AND HDF5_IS_PARALLEL)
  ADD_COMPILE_DEFINITIONS(USE_MPI_HDF5=1)
ELSE()
  ADD_COMPILE_DEFINITIONS(USE_MPI_HDF5=0)
ENDIF()

In the three HDF5 source files, the parallel I/O calls are wrapped with #if USE_MPI_HDF5 instead of the former #if USE_MPI:

! io_hdf5.f90 / hdf5_input.f90 / hdf5_output.f90
#if USE_MPI_HDF5
  CALL H5PSET_FAPL_MPIO_F(Plist_File_ID, communicatorOpt%mpi_val, MPIInfo%mpi_val, iError)
#endif /*USE_MPI_HDF5*/

Fix D — Full path to powershell.exe in userblock step (CMakeLists.txt)

The userblock pre-build step invokes generateuserblock.ps1 via powershell. During the Ninja build, Windows PowerShell is not in the effective PATH. Fixed by using the full path: 

ADD_CUSTOM_COMMAND(TARGET libpiclasstatic PRE_BUILD COMMAND
  "C:/Windows/System32/WindowsPowerShell/v1.0/powershell.exe"
  -ExecutionPolicy Bypass -NonInteractive -File
  "${CMAKE_CURRENT_SOURCE_DIR}/tools/userblock/generateuserblock.ps1" ...)

Fix E — Link MPI libs directly to executables (src/CMakeLists.txt, unitTests/CMakeLists.txt)

On Windows, symbols from a DLL's transitive dependencies are not re-exported through the DLL's import stub (liblibpiclas.dll.a). So mpi_finalize_ and other MPI symbols that libpiclas.dll uses internally are not visible to piclas.exe when it links only against the import stub. Fixed by linking ${linkedlibs} (which contains mpi_f08.o and libmsmpi.dll.a) directly to both the main executable and unit test executables on Windows: 

# src/CMakeLists.txt — add_exec function
FUNCTION(add_exec target)
  ADD_EXECUTABLE(       ${target} ${ARGN})
  TARGET_LINK_LIBRARIES(${target} libpiclasshared)
  IF(WIN32)
    TARGET_LINK_LIBRARIES(${target} ${linkedlibs})
    ...

# unitTests/CMakeLists.txt — add_unit_test function
IF(WIN32)
  TARGET_LINK_LIBRARIES(${target} ${linkedlibs})
ENDIF()

Fix F — Userblock / HDF5 superblock corruption (src/io_hdf5/io_hdf5.f90)

When copy_userblock copies the Fortran-generated userblock to the front of an HDF5 file, it must reserve space before the HDF5 superblock. Without this fix the userblock overwrites the superblock, corrupting the file. The fix calls H5PSET_USERBLOCK_F on the property list before H5FCREATE_F, and passes creation_prp=Plist_File_ID so HDF5 reserves the correctly rounded-up 512-byte-aligned block: 

! io_hdf5.f90 — OpenDataFile, inside the MPIRoot / create-new-file path
IF(userblockSize_loc.GT.0)THEN
  tmp = userblockSize_512
  DO WHILE(tmp.LT.userblockSize_loc)
    tmp = tmp * 2
  END DO
  CALL H5PSET_USERBLOCK_F(Plist_File_ID, tmp, iError)
END IF
CALL H5FCREATE_F(TRIM(FileString), H5F_ACC_TRUNC_F, File_ID, iError, creation_prp=Plist_File_ID)

Fix G — Sequential HDF5 gather-write pattern (src/io_hdf5/hdf5_output.f90)

With USE_MPI=1, USE_MPI_HDF5=0 (MPI compute, serial HDF5), only MPIRoot opens the HDF5 file — non-root ranks have File_ID=0. Calling WriteArrayToHDF5 directly on all ranks would cause H5DCREATE_F(File_ID=0) to fail on ranks 1…N with "Dataset could not be created".

GatheredWriteArray and DistributedWriteArray handle this correctly: each non-root rank sends its local slice to MPIRoot via MPI_GATHERV, and only MPIRoot calls the HDF5 write. The key guard in these routines is: 

! hdf5_output.f90 — GatheredWriteArray
#if USE_MPI && !USE_MPI_HDF5
  ! Gather data from all ranks to MPIRoot, then only MPIRoot writes
  CALL MPI_GATHERV(SendBuf, SendCount, MPI_DOUBLE_PRECISION, &
                   RecvBuf, RecvCounts, Displs, MPI_DOUBLE_PRECISION, &
                   0, MPI_COMM_PICLAS, iError)
  IF(MPIRoot) CALL WriteArrayToHDF5(...)
#else
  CALL WriteArrayToHDF5(...)
#endif

Fix H — WriteDSMCToHDF5 uses GatheredWriteArray (src/particles/dsmc/dsmc_analyze.f90)

WriteDSMCToHDF5 previously opened the DSMCState file with OpenDataFile(single=.FALSE., communicatorOpt=MPI_COMM_PICLAS) and then called WriteArrayToHDF5 directly on all ranks. This bypassed the gather-write pattern and caused "ERROR in WriteArrayToHDF5: Dataset ElemData could not be created" on ranks 1–3.

Fix: removed the OpenDataFile / CloseDataFile wrapper and replaced the direct WriteArrayToHDF5 calls with GatheredWriteArray. Added GatheredWriteArray to the USE MOD_HDF5_Output list: 

! dsmc_analyze.f90 — WriteDSMCToHDF5 (replaces OpenDataFile + WriteArrayToHDF5 + CloseDataFile)
USE MOD_HDF5_Output ,ONLY: WriteAttributeToHDF5, WriteHDF5Header, WriteArrayToHDF5, GatheredWriteArray

ASSOCIATE (&
  nVarX        => INT(nVar+nVar_quality+nVar_HeatPress+nVarMPF,IK) ,&
  PP_nElems    => INT(PP_nElems,IK)                                 ,&
  offsetElem   => INT(offsetElem,IK)                                ,&
  nGlobalElems => INT(nGlobalElems,IK)                           )
  CALL GatheredWriteArray(FileName,create=.FALSE.,DataSetName='ElemData', rank=2,  &
                          nValGlobal=(/nVarX,nGlobalElems/),                        &
                          nVal      =(/nVarX,PP_nElems/),                           &
                          offset    =(/0_IK ,offsetElem/),                          &
                          collective=.false., RealArray=DSMC_MacroVal(:,:))
  IF(SampleElecExcitation) THEN
    CALL GatheredWriteArray(FileName,create=.FALSE.,DataSetName='ExcitationData', rank=2, &
                            nValGlobal=(/nVarExci,nGlobalElems/),                          &
                            nVal      =(/nVarExci,PP_nElems/),                             &
                            offset    =(/0_IK    ,offsetElem/),                            &
                            collective=.false., RealArray=MacroElecExcitation(:,:))
  END IF
END ASSOCIATE

6.4 Running with MPI

# Single process (same as non-MPI)
./build-ucrt64-mpi/bin/piclas.exe parameter.ini

# Multi-process (requires msmpi.dll in PATH)
mpiexec -n 4 ./build-ucrt64-mpi/bin/piclas.exe parameter.ini
Required: disable HDF5 file locking before mpiexec On Windows, HDF5's file-locking mechanism conflicts with how Windows manages file handles across multiple processes. Set this environment variable before running any MPI simulation, otherwise PICLas will abort during the first HDF5 write: 
# MSYS2 / bash
export HDF5_USE_FILE_LOCKING=FALSE
mpiexec -n 4 bin/piclas.exe parameter.ini DSMC.ini

# Windows CMD (.bat file)
set HDF5_USE_FILE_LOCKING=FALSE
mpiexec -n 4 bin\piclas.exe parameter.ini DSMC.ini
This variable is required for all MPI runs (DSMC, PIC, etc.) on Windows. It is not needed for single-process runs.
DSMC cone example (dsmc-cone-3D tutorial) 
export HDF5_USE_FILE_LOCKING=FALSE
export PATH=/c/msys64/ucrt64/bin:$PATH
cd /c/Data/PRJ/Piclas/dsmc-cone-3D
mpiexec -n 4 bin/piclas.exe parameter.ini DSMC.ini
Run time: ~3 minutes for 10 000 iterations on a desktop CPU. Output: DSMCState, DSMCSurfState, and State HDF5 files at each analysis interval.
MPI and PETSc cannot be combined on Windows MSYS2's PETSc is built without MPI (sequential/serial variant). Two separate failures occur:
  • Compile-time: Its headers include petsc/mpiuni/mpiunifdef.h which redefines MPI_WTIME → PETSC_MPI_WTIME. Fortran sees this redefined symbol with no type declaration, causing a fatal compile error in hdg.f90. cmake emits a FATAL_ERROR if both flags are set.
  • Runtime: Even if the header conflict is patched around, PETSc's runtime library calls PetscInitialize() which checks MPI_COMM_WORLD size and aborts with "You are using an MPI-uni (sequential) install of PETSc but trying to launch parallel jobs" when launched with more than one MPI rank. All processes crash with SIGABRT.
Use either LIBS_USE_MPI=ON or LIBS_USE_PETSC=ON, never both.
HDF5 file I/O is serial — gather-write pattern Since MSYS2's HDF5 has no parallel MPI support (USE_MPI_HDF5=0), only MPIRoot (rank 0) opens HDF5 files. Non-root ranks have File_ID=0 and must not call any HDF5 write routines directly. PICLas handles this transparently via the gather-write pattern:
  • Each rank sends its local data slice to MPIRoot via MPI_GATHERV
  • MPIRoot assembles the full array and writes it to HDF5
  • This is done inside GatheredWriteArray and DistributedWriteArray
Any call to WriteArrayToHDF5 directly on all ranks (bypassing the gather-write) will fail on ranks 1–N with "Dataset could not be created". See Fix G and Fix H above.

13. Runtime Bugs (Landau Damping PIC-Poisson Tutorial)

Three Windows-specific runtime bugs were found and fixed while verifying the pic-poisson-landau-damping tutorial with mpiexec -n 4. All three are caused by differences between MS-MPI and the Linux MPI implementations (OpenMPI/MPICH) that PIClas was primarily developed and tested against. None affect single-process runs or Linux.

13.1 MS-MPI MPI_ALLGATHERV(MPI_IN_PLACE) corrupts SideInfo_Shared

Symptom

FATAL ERROR: GlobalSideID not found for Elem …

Occurs immediately during mesh initialisation on every MPI rank.

Root cause

In particle_mesh_readin.f90 a blocking MPI_ALLGATHERV(MPI_IN_PLACE, …) is called on MPI_COMM_LEADERS_SHARED — the communicator that contains only one rank per compute node (the "leader"). On a single compute node this communicator has exactly one member. MS-MPI's implementation of MPI_IN_PLACE on a 1-process communicator zeroes the entire receive buffer before copying from the in-place source — but because the source is the buffer, the source has already been wiped. The result is that SideInfo_Shared columns 1–9 (including SIDE_LOCALID) are all set to zero. A subsequent call to GetGlobalNonUniqueSideID then finds no side with a matching SIDE_LOCALID and aborts.

Why this only affects single-node runs on Windows On a multi-node cluster MPI_COMM_LEADERS_SHARED has more than one member and the MPI_IN_PLACE gather works correctly. On Linux with a 1-process communicator the data is simply left in place (a no-op). MS-MPI is unique in zeroing the buffer first.

The four MPI_IALLGATHERV(MPI_IN_PLACE, …) calls that precede this one (for ElemInfo_Shared, SideInfo_Shared, NodeInfo_Shared, NodeCoords_Shared on MPI_COMM_SHARED with all 4 procs) are not affected: they run on a 4-process communicator where MPI_IN_PLACE works correctly.

Fix — particle_mesh_readin.f90

Guard the blocking gather with a process-count check so it is skipped when there is only one leader (single compute node):

CALL BARRIER_AND_SYNC(SideInfo_Shared_Win,MPI_COMM_SHARED)
IF (myComputeNodeRank.EQ.0) THEN
  ! Only gather across node leaders when there are multiple compute nodes.
  ! MS-MPI MPI_ALLGATHERV(MPI_IN_PLACE,...) on a 1-process communicator corrupts
  ! the receive buffer by zeroing it before filling from the (now-zero) in-place
  ! buffer. With a single node all data is already in shared memory -- skip.
  IF (nLeaderGroupProcs.GT.1) THEN
    CALL MPI_ALLGATHERV(MPI_IN_PLACE,0,MPI_DATATYPE_NULL,SideInfo_Shared,recvcountSide &
        ,displsSide,MPI_STRUCT_SIDE,MPI_COMM_LEADERS_SHARED,IERROR)
  END IF
  CALL MPI_TYPE_FREE(MPI_STRUCT_ELEM,iError)
  CALL MPI_TYPE_FREE(MPI_STRUCT_SIDE,iError)
  CALL MPI_TYPE_FREE(MPI_STRUCT_NODE,iError)
END IF

File: src/particles/particle_mesh/particle_mesh_readin.f90 (around the MPI_ALLGATHERV … MPI_COMM_LEADERS_SHARED block).

13.2 Intersection-based periodic boundary tracking fails

Symptom

ERROR: Particle not inside of Element, ipart …

Occurs during the first time step; particles near x=0 moving in −x direction are removed from the simulation.

Root cause

The default TrackingMethod = refmapping uses intersection-based tracking at periodic boundaries (ParticleBCTracking). On Windows this routine fails to detect when a particle crosses the periodic x=0 face. After the Boris push, particles near x=0 with negative vx end up at x < 0 (outside the domain). SinglePointToElement correctly returns −1, LocateParticleInElement calls RemoveParticle, and the abort fires.

Root cause is likely a floating-point edge case in the intersection test The Linux build with OpenMPI passes this test. The Windows/MS-MPI build triggers a different code path in the intersection geometry, probably due to slightly different floating-point rounding in MSVC-compatible math routines vs. glibc.

Fix — parameter.ini

Add CartesianPeriodic = T immediately after TrackingMethod: 

TrackingMethod              = refmapping
CartesianPeriodic           = T

This activates PeriodicMovement: instead of computing intersection points at periodic faces, it directly wraps the particle position (if pos > xmax: pos = pos − L, if pos < 0: pos = pos + L). PIClas documents this as "Simplified treatment for periodic box with Refmapping. No computation of intersection points at periodic BCs." It is physically equivalent for a Cartesian periodic domain.

With CartesianPeriodic=T and nPeriodicVectors=3, ParticleBCTracking calls PeriodicMovement which wraps positions in all three directions and immediately cycles/returns, bypassing the problematic intersection calculations.

13.3 Single-node shape_function run: MPI exchange halo too small

Symptom

GlobalProcToExchangeProc(EXCHANGE_PROC_RANK,ProcID) is negative.
The halo region might be too small. Try increasing Particles-HaloEpsVelo!

Abort occurs during the particle MPI exchange at the end of the first time step, on a proc that needs to send a particle to a non-adjacent proc.

Root cause — step by step

GlobalProcToExchangeProc maps global proc IDs to local exchange-proc indices. It is built in particle_mpi_halo.f90:IdentifyPartExchangeProcs using MPI_halo_eps as the distance threshold. A proc can only send particles to procs that appear in its exchange list.

With 4 procs on one node and a domain length of 12.566 (proc width ≈ 3.14), the exchange list for each proc is:

Procx rangeExchange listNOT included
00 – 3.14Proc 1 (right), Proc 3 (left via periodic)Proc 2
13.14 – 6.28Proc 0 (left), Proc 2 (right)Proc 3
26.28 – 9.42Proc 1 (left), Proc 3 (right)Proc 0
39.42 – 12.57Proc 2 (left), Proc 0 (right via periodic)Proc 1

The intended behaviour for single-node runs is: MPI_halo_eps = c × dt × SafetyFactor = 1×108 × 0.1 × 1.2 = 1.2×107 — much larger than the domain, so all four procs are in each other's exchange lists.

The code achieves this through the branch in particle_mpi_halo.f90: 

IF (halo_eps.LE.0.) THEN
  ! single-node path: MPI_halo_eps = c * dt * SafetyFactor  (~1.2e7)
ELSE
  ! multi-node path: MPI_halo_eps = halo_eps                (~1.0)
END IF

For single-node runs without shape_function deposition, particle_bgm.f90 leaves halo_eps = 0, and the correct single-node branch is taken.

For single-node runs with PIC-Deposition-Type = shape_function and PIC-shapefunction-dimension = 1, particle_bgm.f90 sets: 

halo_eps = MAX(r_sf, MAX(y_extent, z_extent))
         = MAX(0.5,  MAX(1.0,     1.0     ))
         = 1.0   > 0

Because halo_eps = 1.0 > 0 the code wrongly enters the multi-node branch and sets MPI_halo_eps = 1.0. With a proc width of 3.14 and halo of only 1.0, procs 1 and 3 (and procs 0 and 2) are not in each other's exchange lists. A particle that acquires a large velocity from a corrupted field interpolation can jump across two proc-widths in one time step and then cannot be sent to the target proc → abort.

Why the field interpolation can be corrupted PEM%LocalElemID(iPart) = PEM%GlobalElemID(iPart) − offsetElem. If a misplaced particle carries a GlobalElemID belonging to a different proc, the resulting LocalElemID exceeds nElems and accesses out-of-bounds memory in U_N(LocalElemID)%E, producing a garbage electric field (~1024), a garbage velocity (~−2.7×1023), and a position that wraps incorrectly (integer overflow in CEILING inside PeriodicMovement).

Fix — particle_mpi_halo.f90

Replace the halo_eps-based test with an explicit single-node test. nComputeNodeProcessors and nProcessors_Global are both available in MOD_MPI_Shared_Vars, which is already USEd without an ONLY clause in this subroutine. 

! OLD (incorrect when shape_function sets halo_eps > 0 on a single node):
IF (halo_eps.LE.0.) THEN

! NEW (correctly detects single-node regardless of halo_eps):
IF (nComputeNodeProcessors.EQ.nProcessors_Global) THEN

File: src/particles/particle_mpi/particle_mpi_halo.f90, subroutine IdentifyPartExchangeProcs, the comment block starting "if running on one node, halo_eps is meaningless".

After this fix, single-node runs with shape_function use the single-node path and obtain MPI_halo_eps ≈ 1.2×107, which covers the entire domain and ensures all procs are in each other's exchange lists.

Rebuild required After applying this fix, rebuild PIClas (see Section 6) and copy the new piclas.exe to the simulation directory before re-running.
