Overview

PICLas is a Particle-in-Cell / Direct Simulation Monte Carlo (PIC-DSMC) plasma simulation code written in Fortran/C. It was originally developed for Linux/HPC environments. This guide documents all steps taken to build and run PICLas on Windows 11 using the MSYS2 UCRT64 environment, including source code patches, build system configuration, and runtime fixes.

Key strategy PICLas is compiled natively on Windows using GCC/GFortran provided by MSYS2 (not Cygwin, not WSL). The resulting piclas-win.exe is a real Windows executable that runs without MSYS2 at runtime, provided the required DLLs are bundled (see Distribution).
Agent tooling on this workstation — use PowerShell, not the Bash tool When working in this repo with an AI coding agent (Claude Code etc.), the Bash tool returns empty output on this Windows machine ("Bash completed with no output") even when the command succeeds, and background (run_in_background) Bash invocations may silently not execute — e.g. a reggie recheck launched via the Bash tool produced no log and exited 0, while the identical script run through PowerShell worked immediately. Drive everything through the PowerShell tool. When a POSIX shell is required (running the .sh build/reggie scripts, the MSYS2 toolchain), invoke MSYS2 bash from PowerShell and redirect output to a file you then read back: 
$env:PATH = "C:\msys64\ucrt64\bin;C:\msys64\usr\bin;" + $env:PATH
& C:\msys64\usr\bin\bash.exe "C:\Data\PRJ\piclas-win\run_nig_all.sh" *> "C:\Data\PRJ\piclas-win\run_console.log"
Do this from the first command of each session to avoid silently lost output and non-running background jobs.

4. Source Code Changes

PICLas requires four modifications to build and run correctly on Windows. These changes are backward-compatible: on Linux/macOS the new C file compiles to a no-op stub and behaviour is unchanged.

4.1 Windows Wildcard Expansion â€” glob_windows.c NEW FILE

On Linux, the shell expands wildcards like plasma_wave_State_*.h5 before passing them to the program. Windows shells (CMD, PowerShell) do not â€” the literal string * reaches the program unchanged. This new file implements wildcard expansion using the Win32 API so that piclas2vtk parameter.ini State_*.h5 works from any Windows terminal.

File: piclas-master/src/globals/glob_windows.c 
// Win32 glob expansion â€” called from Fortran via ISO_C_BINDING
// On non-Windows: stub always returns 0 (shell already expands globs)
#ifdef _WIN32
#include <windows.h>
#include <string.h>
#include <stdlib.h>

static int cmp_str(const void *a, const void *b) {
    return strcmp(*(const char **)a, *(const char **)b);
}

int glob_expand_c(const char *pattern, char *results, int results_len) {
    WIN32_FIND_DATAA fd;
    HANDLE h;
    char *ptrs[4096];
    int count = 0;
    int pos = 0;

    h = FindFirstFileA(pattern, &fd);
    if (h == INVALID_HANDLE_VALUE) return 0;

    /* Determine directory prefix from pattern */
    char dir[MAX_PATH] = "";
    const char *slash = strrchr(pattern, '/');
    const char *bslash = strrchr(pattern, '\\');
    const char *last = slash > bslash ? slash : bslash;
    if (last) { int n = (int)(last - pattern) + 1; memcpy(dir, pattern, n); dir[n] = 0; }

    do {
        if (count >= 4096) break;
        char *entry = malloc(MAX_PATH);
        snprintf(entry, MAX_PATH, "%s%s", dir, fd.cFileName);
        ptrs[count++] = entry;
    } while (FindNextFileA(h, &fd));
    FindClose(h);

    qsort(ptrs, count, sizeof(char *), cmp_str);

    for (int i = 0; i < count; i++) {
        int len = strlen(ptrs[i]);
        if (pos + len + 1 >= results_len) { free(ptrs[i]); break; }
        memcpy(results + pos, ptrs[i], len + 1);
        pos += len + 1;
        free(ptrs[i]);
    }
    return count;
}
#else
int glob_expand_c(const char *pattern, char *results, int results_len) {
    (void)pattern; (void)results; (void)results_len;
    return 0;  /* Linux/macOS: shell handles glob expansion */
}
#endif

4.2 Fortran Command-Line Interface â€” commandlinearguments.f90 MODIFIED

Add an ISO_C_BINDING interface for glob_expand_c and call it for any argument containing * or ?.

File: piclas-master/src/globals/commandlinearguments.f90

Add at the top of the module, inside the relevant subroutine:

USE ISO_C_BINDING, ONLY: C_INT, C_CHAR, C_NULL_CHAR

INTERFACE
  FUNCTION glob_expand_c(pattern, results, results_len) BIND(C, NAME='glob_expand_c')
    USE ISO_C_BINDING, ONLY: C_INT, C_CHAR
    CHARACTER(KIND=C_CHAR), INTENT(IN)        :: pattern(*)
    CHARACTER(KIND=C_CHAR), INTENT(OUT)       :: results(*)
    INTEGER(KIND=C_INT),    VALUE, INTENT(IN) :: results_len
    INTEGER(KIND=C_INT)                       :: glob_expand_c
  END FUNCTION glob_expand_c
END INTERFACE

Then wrap each argument that contains * or ? through glob_expand_c before storing it.

5. Build Fixes

5.1 TEMP/TMP environment variables

On some Windows systems, the TEMP and TMP variables point to C:\Windows\Temp (a system directory). GCC cannot write temporary files there and aborts with "Cannot create temporary file in C:\WINDOWS\: Permission denied".

Set them to the user temp directory before building. Add these lines to any build script:

export TEMP='/c/Users/YOUR_USERNAME/AppData/Local/Temp'
export TMP=$TEMP

Or in a .bat file, use /tmp which MSYS2 maps to C:\msys64\tmp:

"%BASH%" -c "export TEMP=/tmp && export TMP=/tmp && cmake --build ..."

5.2 Missing Fortran module include directory

GFortran warns about C:/msys64/ucrt64/mod/static not existing. The directory is referenced by the HDF5 Fortran CMake configuration for static library modules. The warning is harmless but can be silenced by creating the empty directory: 

mkdir -p /c/msys64/ucrt64/mod/static

5.3 PETSc pkg-config wrapper FIX

MSYS2 installs PETSc with variant-specific .pc files (petsc-dso.pc, petsc-cso.pc, etc.) but not a plain petsc.pc. PICLas's FindPETSc.cmake uses pkg-config to search for the package name petsc and, when it fails to find it, falls back to downloading and compiling PETSc from source â€” which fails on Windows because the PETSc configure script uses Unix environment-variable syntax incompatible with cmd.exe.

The fix is to create a plain petsc.pc wrapper pointing to the dso variant (double precision, serial, optimized): 

# Create file: C:\msys64\ucrt64\lib\pkgconfig\petsc.pc
prefix=/c/msys64/ucrt64
exec_prefix=${prefix}
libdir=${exec_prefix}/lib/petsc/dso
includedir=${exec_prefix}/include/petsc

Name: PETSc
Description: Sequential double precision PETSc build
Version: 3.24.5
Requires.private: hwloc openblas
Cflags: -I${includedir}/dso -I${includedir}
Libs: -L${libdir} -lpetsc
Libs.private: -L${libdir} -lpetsc -lgfortran -lquadmath

Create this file from the MSYS2 shell:

cat > /c/msys64/ucrt64/lib/pkgconfig/petsc.pc << 'EOF'
prefix=/c/msys64/ucrt64
exec_prefix=${prefix}
libdir=${exec_prefix}/lib/petsc/dso
includedir=${exec_prefix}/include/petsc

Name: PETSc
Description: Sequential double precision PETSc build
Version: 3.24.5
Requires.private: hwloc openblas
Cflags: -I${includedir}/dso -I${includedir}
Libs: -L${libdir} -lpetsc
Libs.private: -L${libdir} -lpetsc -lgfortran -lquadmath
EOF

Verify it works:

pkg-config --modversion petsc
# Expected output: 3.24.5

When building with PETSc, also pass these cmake flags to ensure the system library is used rather than triggering a source build:

-DLIBS_BUILD_PETSC=OFF \
-DPETSC_LINK_LIBRARIES=C:/msys64/ucrt64/lib/petsc/dso/libpetsc.dll.a

And set the environment variable so FindPETSc.cmake finds the pkgconfig directory:

export PETSC_DIR=/c/msys64/ucrt64

5.4 Parallel build race condition (unit tests)

When Ninja builds in parallel, unit test source files can start compiling before the Fortran .mod files in libpiclasstaticF90 have been generated, causing "Fatal Error: Cannot open module file 'mod_globals.mod'".

Workaround: build with --target piclas to skip unit tests, or set PICLAS_UNITTESTS=OFF. The CMake dependency fix in unitTests/CMakeLists.txt (Section 4.3) also mitigates this.

6. MPI Build (MS-MPI)

PICLas can be built with MPI support on Windows using Microsoft MPI (MS-MPI v10.1) provided by MSYS2. This enables multi-process parallel simulations via mpiexec -n N piclas-win.exe. Note that MSYS2's HDF5 is built without parallel MPI support, so file I/O remains serial even in MPI runs â€” this is the standard behaviour for most desktop/workstation use cases.

6.1 Install MS-MPI Package

Install the MSYS2 MS-MPI wrapper package (provides mpicc, mpifort, import library libmsmpi.dll.a, and header mpi.h):

pacman -S --noconfirm mingw-w64-ucrt-x86_64-msmpi
MS-MPI runtime The MSYS2 package provides the import library for linking. For running MPI programs you also need the MS-MPI runtime DLL (msmpi.dll). This is provided by Microsoft MPI v10 and is typically already installed as part of the MSYS2 package or can be installed separately.

6.2 Source Code and CMake Fixes

Eight fixes are needed to get the MPI build working correctly on Windows. These are all already applied in the patched source tree.

Fix A â€” mpi_f08.o linker ordering (cmake/SetLibraries.cmake)

MS-MPI v10 provides only the legacy MODULE MPI interface. PICLas ships a compatibility wrapper cmake/mpi_f08.f90 that provides the modern typed MPI_F08 interface, compiled into mpi_f08.o. GNU ld scans archives left-to-right: the object file must appear before libmsmpi.dll.a so that its undefined references (mpi_wtick_ etc.) are known when the import library is scanned. Changed from LIST(APPENDâ€¦) to LIST(INSERT â€¦ 0 â€¦): 

# mpi_f08.o must precede libmsmpi.dll.a in the link command
IF(EXISTS "${_mpi_f08_dir}/mpi_f08.o")
  LIST(INSERT linkedlibs 0 "${_mpi_f08_dir}/mpi_f08.o")
ENDIF()

Fix B â€” HDF5 parallel detection false positive (cmake/SetLibraries.cmake)

MSYS2 HDF5 is built without parallel MPI support (HDF5_PROVIDES_PARALLEL=OFF in hdf5-config.cmake). The original detection used grep mpio_f which matched the symbol h5fd_mpio_f_ (an MPIO file-driver constant that is always present), causing a false HDF5_IS_PARALLEL=TRUE. Changed to search for the specific function that only exists in parallel-capable builds:

ELSEIF(WIN32)
  EXECUTE_PROCESS(COMMAND nm -g ${HDF5_Fortran_LIBRARY_hdf5_fortran}
                  COMMAND grep h5pset_fapl_mpio_f

Fix C â€” USE_MPI_HDF5 compile flag + source guards (cmake/SetLibraries.cmake, 3 HDF5 source files)

Fix E â€” Link MPI libs directly to executables (src/CMakeLists.txt, unitTests/CMakeLists.txt)

Fix F â€” Userblock / HDF5 superblock corruption (src/io_hdf5/io_hdf5.f90)

When copy_userblock copies the Fortran-generated userblock to the front of an HDF5 file, it must reserve space before the HDF5 superblock. Without this fix the userblock overwrites the superblock, corrupting the file. The fix calls H5PSET_USERBLOCK_F on the property list before H5FCREATE_F, and passes creation_prp=Plist_File_ID so HDF5 reserves the correctly rounded-up 512-byte-aligned block: 

! io_hdf5.f90 â€” OpenDataFile, inside the MPIRoot / create-new-file path
IF(userblockSize_loc.GT.0)THEN
  tmp = userblockSize_512
  DO WHILE(tmp.LT.userblockSize_loc)
    tmp = tmp * 2
  END DO
  CALL H5PSET_USERBLOCK_F(Plist_File_ID, tmp, iError)
END IF
CALL H5FCREATE_F(TRIM(FileString), H5F_ACC_TRUNC_F, File_ID, iError, creation_prp=Plist_File_ID)

Fix G â€” Sequential HDF5 gather-write pattern (src/io_hdf5/hdf5_output.f90)

With USE_MPI=1, USE_MPI_HDF5=0 (MPI compute, serial HDF5), only MPIRoot opens the HDF5 file â€” non-root ranks have File_ID=0. Calling WriteArrayToHDF5 directly on all ranks would cause H5DCREATE_F(File_ID=0) to fail on ranks 1â€¦N with "Dataset could not be created".

GatheredWriteArray and DistributedWriteArray handle this correctly: each non-root rank sends its local slice to MPIRoot via MPI_GATHERV, and only MPIRoot calls the HDF5 write. The key guard in these routines is: 

! hdf5_output.f90 â€” GatheredWriteArray
#if USE_MPI && !USE_MPI_HDF5
  ! Gather data from all ranks to MPIRoot, then only MPIRoot writes
  CALL MPI_GATHERV(SendBuf, SendCount, MPI_DOUBLE_PRECISION, &
                   RecvBuf, RecvCounts, Displs, MPI_DOUBLE_PRECISION, &
                   0, MPI_COMM_PICLAS, iError)
  IF(MPIRoot) CALL WriteArrayToHDF5(...)
#else
  CALL WriteArrayToHDF5(...)
#endif

Fix H â€” WriteDSMCToHDF5 uses GatheredWriteArray (src/particles/dsmc/dsmc_analyze.f90)

WriteDSMCToHDF5 previously opened the DSMCState file with OpenDataFile(single=.FALSE., communicatorOpt=MPI_COMM_PICLAS) and then called WriteArrayToHDF5 directly on all ranks. This bypassed the gather-write pattern and caused "ERROR in WriteArrayToHDF5: Dataset ElemData could not be created" on ranks 1â€“3.

                          nValGlobal=(/nVarX,nGlobalElems/),                        &

                          offset    =(/0_IK ,offsetElem/),                          &

  IF(SampleElecExcitation) THEN

MPI and PETSc cannot be combined on Windows MSYS2's PETSc is built without MPI (sequential/serial variant). Two separate failures occur:
  • Compile-time: Its headers include petsc/mpiuni/mpiunifdef.h which redefines MPI_WTIME â†’ PETSC_MPI_WTIME. Fortran sees this redefined symbol with no type declaration, causing a fatal compile error in hdg.f90. cmake emits a FATAL_ERROR if both flags are set.
  • Runtime: Even if the header conflict is patched around, PETSc's runtime library calls PetscInitialize() which checks MPI_COMM_WORLD size and aborts with "You are using an MPI-uni (sequential) install of PETSc but trying to launch parallel jobs" when launched with more than one MPI rank. All processes crash with SIGABRT.
Use either LIBS_USE_MPI=ON or LIBS_USE_PETSC=ON, never both.
HDF5 file I/O is serial â€” gather-write pattern Since MSYS2's HDF5 has no parallel MPI support (USE_MPI_HDF5=0), only MPIRoot (rank 0) opens HDF5 files. Non-root ranks have File_ID=0 and must not call any HDF5 write routines directly. PICLas handles this transparently via the gather-write pattern:
  • Each rank sends its local data slice to MPIRoot via MPI_GATHERV
  • MPIRoot assembles the full array and writes it to HDF5
  • This is done inside GatheredWriteArray and DistributedWriteArray
Any call to WriteArrayToHDF5 directly on all ranks (bypassing the gather-write) will fail on ranks 1â€“N with "Dataset could not be created". See Fix G and Fix H above.

Four Windows-specific runtime bugs were found and fixed while verifying the pic-poisson-landau-damping tutorial with mpiexec -n 4. All four are caused by differences between MS-MPI and the Linux MPI implementations (OpenMPI/MPICH) that PIClas was primarily developed and tested against. None affect single-process runs or Linux.

13.1 MS-MPI MPI_ALLGATHERV(MPI_IN_PLACE) corrupts SideInfo_Shared

Symptom

FATAL ERROR: GlobalSideID not found for Elem â€¦

Occurs immediately during mesh initialisation on every MPI rank.

Root cause

In particle_mesh_readin.f90 a blocking MPI_ALLGATHERV(MPI_IN_PLACE, â€¦) is called on MPI_COMM_LEADERS_SHARED â€” the communicator that contains only one rank per compute node (the "leader"). On a single compute node this communicator has exactly one member. MS-MPI's implementation of MPI_IN_PLACE on a 1-process communicator zeroes the entire receive buffer before copying from the in-place source â€” but because the source is the buffer, the source has already been wiped. The result is that SideInfo_Shared columns 1â€“9 (including SIDE_LOCALID) are all set to zero. A subsequent call to GetGlobalNonUniqueSideID then finds no side with a matching SIDE_LOCALID and aborts.

Why this only affects single-node runs on Windows On a multi-node cluster MPI_COMM_LEADERS_SHARED has more than one member and the MPI_IN_PLACE gather works correctly. On Linux with a 1-process communicator the data is simply left in place (a no-op). MS-MPI is unique in zeroing the buffer first.
The four preceding MPI_IALLGATHERV(MPI_IN_PLACE, â€¦) calls are also affected StartCommunicateMeshReadin posts four non-blocking gathers (ElemInfo_Shared, SideInfo_Shared, NodeInfo_Shared, NodeCoords_Shared) on MPI_COMM_LEADERS_SHARED â€” the same 1-process communicator that triggers the MS-MPI zeroing bug. This is documented as Bug #4 below (Â§13.4). The fix guards both the non-blocking gathers in StartCommunicateMeshReadin and the corresponding MPI_WAITALL / MPI_TYPE_FREE calls in FinishCommunicateMeshReadin.

13.2 Intersection-based periodic boundary tracking fails

Symptom

ERROR: Particle not inside of Element, ipart â€¦

Occurs during the first time step; particles near x=0 moving in âˆ’x direction are removed from the simulation.

Root cause

The default TrackingMethod = refmapping uses intersection-based tracking at periodic boundaries (ParticleBCTracking). On Windows this routine fails to detect when a particle crosses the periodic x=0 face. After the Boris push, particles near x=0 with negative vx end up at x < 0 (outside the domain). The Windows/MS-MPI build triggers a different code path in the intersection geometry, probably due to slightly different floating-point rounding in MSVC-compatible math routines vs. glibc.

Fix â€” parameter.ini

Add CartesianPeriodic = T immediately after TrackingMethod:

CartesianPeriodic           = T

What SetGPU.cmake does

  1. Runs FIND_PACKAGE(CUDAToolkit 11.0 REQUIRED) to locate nvcc and CUDA libraries.
  2. Searches for cl.exe (MSVC host compiler) in known VS 2019/2022 install paths and on PATH; sets CMAKE_CUDA_HOST_COMPILER.
  3. Pre-sets CMAKE_CUDA_COMPILER_WORKS=TRUE and CMAKE_CUDA_ABI_COMPILED=TRUE to bypass CMake's CUDA link test (which uses MSVC-style flags incompatible with MinGW's linker).
  4. Calls ENABLE_LANGUAGE(CUDA).
  5. Replaces CMAKE_CUDA_CREATE_STATIC_LIBRARY with the GNU ar qc rule (CMake's CUDA default is MSVC-style).
  6. Clears CMAKE_CUDA_IMPLICIT_LINK_LIBRARIES and CMAKE_CUDA_IMPLICIT_LINK_DIRECTORIES (MSVC bare library names are incompatible with MinGW's ld).
  7. Restores CMAKE_LINK_LIBRARY_FLAG="-l" and CMAKE_LIBRARY_PATH_FLAG="-L" (CUDA language detection sets these to MSVC style globally).

What src/CMakeLists.txt does for GPU

  1. Adds gpu_vars.f90 and gpu_interface.f90 to the Fortran source list.
  2. Compiles all src/gpu/*.cu files into the piclasGPU static library with CUDA_RUNTIME_LIBRARY=None and CUDA_SEPARABLE_COMPILATION=OFF.
  3. Sets CUDA include directories to only src/gpu/ (prevents MinGW UCRT64 headers from reaching nvcc/cl.exe).
  4. Passes -Xcompiler=/GS-,/MD,/Zc:threadSafeInit- to cl.exe via nvcc.
  5. Runs patch_cuda_comdat.py on each .cu.obj file as a PRE_LINK step.
  6. At configure time, generates libcudart.dll.a (MinGW import library) from cudart64_13.dll using gendef + dlltool.
  7. Compiles piclasGPUStubs (MinGW-native C stubs for MSVC runtime symbols).
  8. Links piclasGPU, piclasGPUStubs, and libcudart.dll.a into libpiclasshared and piclas-win.exe.
  9. Runs objcopy --remove-section=.voltbl on libpiclas.dll after linking.

14.5 Windows/CUDA Technical Challenges and Solutions

Combining nvcc (MSVC host ABI) with MinGW/gfortran (GNU ABI) on Windows involves a series of non-obvious problems. Each is documented here so future maintainers understand why each workaround exists.

Challenge 1 â€” COMDAT .bss sections cause ACCESS VIOLATION at DLL load

Symptom: libpiclas.dll fails to load with STATUS_DLL_INIT_FAILED (error 1114) at import time.
Root cause: nvcc (using cl.exe as host) emits two function-local statics as LINK_ONCE_DISCARD COMDAT .bss sections in each .cu.obj: 

?__ref@?1??____nv_dummy_param_ref@@YAXPEAX@Z@4PEAPECXEA
?__ref@?2??__nv_cudaEntityRegisterCallback@@YAXPEAPEAX@Z@4PEAPECXEA

GNU ld discards COMDAT sections, leaving IMAGE_REL_AMD64_REL32 relocations in ?__sti____cudaRegisterAll with displacement zero. This produces mov %rax, 0(%rip) â€” writing the CUDA fat-binary handle into the read-only .text section â†’ access violation.
Fix: patch_cuda_comdat.py clears IMAGE_SCN_LNK_COMDAT from every .bss section in the COFF object AND zeroes the COMDAT Selection byte in the matching auxiliary symbol record. GNU ld then treats them as regular BSS and always includes them.

Challenge 2 â€” cudart.lib MSVC static loader causes DLL init ordering crash

Symptom: STATUS_DLL_INIT_FAILED even after COMDAT fix, or intermittent CUDA API failures on first call.
Root cause: NVIDIA's cudart.lib contains cudart_static_loader.obj â€” an MSVC-compiled .CRT$XCU static initializer that calls LoadLibrary("cudart64_13.dll"). Under GNU ld, .CRT$XCU ordering is not guaranteed; the fat-binary registration constructors in piclasGPU.a may run before cudart64_13.dll is loaded.
Fix: At configure time, SetGPU.cmake runs gendef and dlltool to generate a MinGW-native import library libcudart.dll.a from cudart64_13.dll. This puts cudart64_13.dll in the PE IAT (import address table), so Windows loads it before any DllMain or constructor runs â€” correct ordering without relying on .CRT$XCU indices. 

# cmake/SetGPU.cmake generates these commands at configure time:
gendef  C:/Program Files/NVIDIA GPU Computing Toolkit/CUDA/v12.x/bin/x64/cudart64_13.dll
dlltool -d cudart64_13.def -l libcudart.dll.a -D cudart64_13.dll

Challenge 3 â€” nvcc requires MSVC cl.exe (MinGW g++ unsupported)

nvcc on Windows does not support MinGW g++ as a host compiler ("unsupported target OS" error). SetGPU.cmake searches for cl.exe in known VS 2019/2022 install paths and sets CMAKE_CUDA_HOST_COMPILER before ENABLE_LANGUAGE(CUDA). The extern "C" interface in piclas_gpu.h ensures the MSVC-ABI CUDA objects link correctly against the MinGW-compiled Fortran code.

Challenge 4 â€” CMake CUDA link test fails with MinGW

CMake's ENABLE_LANGUAGE(CUDA) runs a test link using cmake -E vs_link_exe (MSVC protocol) because the host compiler is cl.exe. This drives MinGW's ld with MSVC flags (/nologo, /out:) and windres.exe with MSVC manifest arguments â€” both fail.
Fix: Pre-set CMAKE_CUDA_COMPILER_WORKS=TRUE and CMAKE_CUDA_ABI_COMPILED=TRUE in the cache before ENABLE_LANGUAGE(CUDA). CMake skips the link test and ABI detection, treating them as already resolved. Since piclasGPU is a static library, no CUDA executable link is ever needed during the build.

Challenge 5 â€” GNU ar rule overwritten by CUDA language detection

After ENABLE_LANGUAGE(CUDA), CMake sets CMAKE_CUDA_CREATE_STATIC_LIBRARY to an MSVC-style rule ("ar.exe /nologo /out:TARGET OBJECTS"). MinGW ar does not understand /nologo or /out:.
Fix: Immediately after ENABLE_LANGUAGE(CUDA), override: 

SET(CMAKE_CUDA_CREATE_STATIC_LIBRARY
  "<CMAKE_AR> qc <TARGET> <LINK_FLAGS> <OBJECTS>"
  CACHE STRING "GNU ar rule for CUDA static library" FORCE)

Challenge 6 â€” CUDA implicit link libraries use MSVC bare names

ENABLE_LANGUAGE(CUDA) probes the MSVC toolchain and records all implicit Windows SDK libraries (kernel32, ws2_32, cudart, etc.) as bare names into CMAKE_CUDA_IMPLICIT_LINK_LIBRARIES. MinGW's ld requires -l<name> â€” bare names are unresolvable.
Fix: Clear both variables immediately after language setup: 

SET(CMAKE_CUDA_IMPLICIT_LINK_LIBRARIES   "" CACHE INTERNAL "")
SET(CMAKE_CUDA_IMPLICIT_LINK_DIRECTORIES "" CACHE INTERNAL "")

Challenge 7 â€” CUDA language detection clobbers global linker flags

Including Platform/Windows-NVIDIA-CUDA.cmake pulls in Platform/Windows-MSVC.cmake, which sets CMAKE_LINK_LIBRARY_FLAG="" (empty, MSVC style) and CMAKE_LIBRARY_PATH_FLAG="-LIBPATH:" globally. This strips the -l prefix from all subsequent gfortran/ld link commands â€” HDF5, MPI, and PETSc all stop linking.
Fix: 

SET(CMAKE_LINK_LIBRARY_FLAG "-l")
SET(CMAKE_LIBRARY_PATH_FLAG "-L")
SET(CMAKE_CUDA_LINKER_PREFERENCE_PROPAGATES 0)

Challenge 8 â€” .voltbl section makes libpiclas.dll malformed

MSVC-compiled CUDA objects contain a .voltbl section (volatile-metadata / Control-Flow-Guard table). GNU ld assigns it an address outside the normal DLL address range, producing STATUS_INVALID_IMAGE_FORMAT â€” Windows refuses to load the DLL.
Fix: Strip the section post-link with objcopy: 

C:/msys64/ucrt64/bin/objcopy.exe --remove-section=.voltbl libpiclas.dll

This is an automatic CMake POST_BUILD step on the libpiclasshared target.

Challenge 9 â€” MSVC runtime symbols undefined in MinGW link

CUDA objects compiled with cl.exe (and NVIDIA's pre-compiled cudart.lib) reference MSVC-specific symbols that do not exist in MinGW's libraries: __security_cookie, __security_check_cookie, __GSHandlerCheck, __local_stdio_printf_options, _Init_thread_epoch, _Init_thread_header, _Init_thread_footer.
Fix: src/gpu/msvc_stubs.c compiled by MinGW gcc provides inert no-op stubs for all of these. It is built into piclasGPUStubs.a and linked with --whole-archive to guarantee all stubs are present.

Challenge 10 â€” api-ms-win-crt-private dependency appears in DLL import table

MSVC-compiled objects in cudart.lib (compiled with /MD) reference basic CRT functions (memcpy, memcmp, strchr, etc.) via MSVC DLL import thunks: call [__imp_memcpy]. GNU ld creates IAT entries for api-ms-win-crt-private-l1-1-0.dll, which is in System32\downlevel\ (not on the normal DLL search path) â†’ runtime DLL-not-found failure.
Fix: msvc_stubs.c defines __imp_memcpy, __imp_memcmp, etc. as function pointers to MinGW's own implementations. GNU ld resolves the MSVC relocations to these local pointer variables â€” no IAT entry is created and the private UCRT DLL dependency disappears entirely.

Challenge 11 â€” Thread-safe static init stubs with <<<>>> syntax

CUDA <<<>>> kernel launch syntax causes nvcc to generate a host-side stub containing a function-local static with MSVC thread-safe initialization (_Init_thread_header / _Init_thread_footer / _Init_thread_epoch). Even with /Zc:threadSafeInit-, some stubs remain.
Fix: particle_push.cu uses the explicit cudaLaunchKernel() API instead of <<<>>>: 

void *args[] = { &d_PartState, &d_isActive, &nPart, &dt };
cudaLaunchKernel((const void*)particle_push_kernel, grid, block, args, 0, nullptr);

Challenge 12 â€” CMAKE_CUDA_CREATE_SHARED_LIBRARY override silently ignored (31 LNK2019 on __imp_fminf / C99 float math)

Symptom: bin\libpiclasGPU.dll fails to link with 31 LNK2019 errors on __imp_fminf, __imp_fmaxf, __imp_exp2f, __imp_cbrtf, __imp_erff, __imp__dsign, and 25 more C99 float/double math symbols. The linker warning LNK4098: MSVCRT conflicts with other libraries also appears. The build log shows nvcc -shared -cudart shared ... â€” but the CMAKE_CUDA_CREATE_SHARED_LIBRARY override in src/CMakeLists.txt specifies -cudart none. The override is not being applied.

Root cause (two layers):

  1. CMake's CUDA platform module (loaded during ENABLE_LANGUAGE(CUDA) in cmake/SetGPU.cmake) stores CMAKE_CUDA_CREATE_SHARED_LIBRARY as a CACHE INTERNAL variable â€” with FORCE. A subsequent plain SET(CMAKE_CUDA_CREATE_SHARED_LIBRARY ...) without FORCE cannot override a cached value and is silently ignored, even in a fresh configure. The platform default (-cudart shared with no /DEFAULTLIB flags) survives into the generated Ninja rule.
  2. With -cudart shared, nvcc auto-links cudadevrt.lib, which injects /DEFAULTLIB:LIBCMT (static MSVC CRT). The MSVC linker then resolves UCRT symbols through libucrt.lib (static UCRT) rather than ucrt.lib (dynamic UCRT import library). libucrt.lib does not expose __imp_fminf / __imp_fmaxf / â€¦ DLL-import stubs â€” those live only in ucrt.lib â€” so every C99 float math overload that CUDA's device wrapper headers generate becomes an unresolved external.

Fix: Add CACHE INTERNAL "" FORCE to the SET(CMAKE_CUDA_CREATE_SHARED_LIBRARY ...) call in src/CMakeLists.txt so it runs after the platform module's forced SET and wins: 

SET(CMAKE_CUDA_CREATE_SHARED_LIBRARY
    "<CMAKE_CUDA_COMPILER> -shared -cudart none -ccbin \"...\" -o <TARGET> <OBJECTS>
     -Xlinker /LIBPATH:\"...\" -Xlinker cudart.lib
     -Xlinker /NODEFAULTLIB:LIBCMT -Xlinker /NODEFAULTLIB:LIBCMTD
     -Xlinker /NODEFAULTLIB:libucrt -Xlinker /DEFAULTLIB:ucrt"
    CACHE INTERNAL "CUDA SHARED library link rule (Windows MinGW+MSVC host)" FORCE)

With -cudart none the automatic cudadevrt/cudart library injection is suppressed; cudart.lib is added explicitly instead. /NODEFAULTLIB:LIBCMT and /NODEFAULTLIB:libucrt remove the static-CRT defaults; /DEFAULTLIB:ucrt adds the dynamic UCRT import library which provides all 31 missing __imp_* stubs. After applying this fix a fresh cmake --preset windows-ucrt64-gpu (reconfigure) is required so CMake regenerates build.ninja with the corrected link rule.

Secondary failure after applying the CACHE FORCE fix â€” LNK1181 on "Files\NVIDIA.obj"
Once the template is applied, a second error can appear: LINK : fatal error LNK1181: cannot open input file 'Files\NVIDIA.obj'. This is caused by the /LIBPATH: argument containing spaces (C:/Program Files/NVIDIA GPU Computing Toolkit/â€¦). When Ninja wraps the link rule in cmd.exe /C "â€¦", the inner quotes around /LIBPATH:"â€¦" are consumed by cmd.exe's quote parser. nvcc receives the argument with its quotes stripped and passes it to link.exe without re-quoting; link.exe then splits the path at the first space and treats Files\NVIDIA as a stray object file.

Fix (already in src/CMakeLists.txt): at configure time, copy cudart.lib from the CUDA toolkit into the build directory (${CMAKE_BINARY_DIR}/lib/cuda/), which is assumed to be space-free (e.g. C:/Data/PRJ/piclas-win/dsmc-cone-2D/lib/cuda). Then use that path in /LIBPATH: â€” no quotes needed: 
SET(_CUDA_LOCAL_LIB_DIR "${CMAKE_BINARY_DIR}/lib/cuda")
FILE(MAKE_DIRECTORY "${_CUDA_LOCAL_LIB_DIR}")
FILE(COPY "${CUDAToolkit_LIBRARY_DIR}/cudart.lib" DESTINATION "${_CUDA_LOCAL_LIB_DIR}")
# Template uses space-free build-dir path â€” no quoting required
-Xlinker /LIBPATH:${_CUDA_LOCAL_LIB_DIR} -Xlinker cudart.lib
CMake's FILE(COPY ...) runs at configure time, so cudart.lib is present in the build tree before Ninja starts. During configure you should see: -- CUDA cudart.lib staged: .../dsmc-cone-2D/lib/cuda/cudart.lib
Note â€” PowerShell GetShortPathName() approach was tried first
An earlier attempt used [System.IO.Path]::GetShortPathName() to obtain the 8.3 form of the path. This failed because PowerShell may not be in the cmake subprocess's PATH, or 8.3 names may be disabled on the volume. The copy-to-build-dir approach is unconditionally reliable as long as the build directory itself has no spaces.

14.8 GPU Init / Finalize Call Sites

GPU_Init and GPU_Finalize are called inside the particle initialization and finalization routines using Fortran's BLOCK construct, which scopes the USE statements to avoid polluting the module's namespace.

Initialization â€” src/particles/particle_init.f90, InitParticles()

Called at the end of InitParticles, after PDM%maxParticleNumber has been set and all particle arrays (PartState, PDM%ParticleInside, etc.) have been allocated:

#if PICLAS_USE_GPU
! Initialize GPU: locate device, allocate device buffers for all particles
BLOCK
  USE MOD_GPU_Interface, ONLY: GPU_Init
  USE MOD_Particle_Vars, ONLY: PDM
  CALL GPU_Init(PDM%maxParticleNumber)
END BLOCK
#endif /*PICLAS_USE_GPU*/

ParticlesInitIsDone=.TRUE.

GPU_Init calls piclas_gpu_init() (prints device info), piclas_gpu_alloc_buffers(nMaxPart) (allocates device-side PartState + isActive arrays), and GPU_AllocActiveMask(nMaxPart) (allocates the host-side integer mask buffer). After this call, GPUInitialized = .TRUE..

Finalization â€” src/particles/particle_init.f90, FinalizeParticles()

Called at the end of FinalizeParticles, before the subroutine returns. The GPUInitialized guard prevents double-free if GPU was never initialized (e.g., non-GPU builds that somehow reach this path):

#if PICLAS_USE_GPU
! Finalize GPU: free device buffers and reset CUDA device
BLOCK
  USE MOD_GPU_Interface, ONLY: GPU_Finalize
  USE MOD_GPU_Vars,      ONLY: GPUInitialized
  IF (GPUInitialized) CALL GPU_Finalize()
END BLOCK
#endif /*PICLAS_USE_GPU*/

GPU_Finalize frees the host mask buffer, calls piclas_gpu_free_buffers() (frees device arrays), then piclas_gpu_finalize() (cudaDeviceReset()). Sets GPUInitialized = .FALSE..

14.9 DSMC Time-Step Integration

The GPU batch push is integrated into src/timedisc/timedisc_TimeStep_DSMC.f90 (PP_TimeDiscMethod==4) using three coordinated changes.

Step A â€” Module imports (top of subroutine)

#if PICLAS_USE_GPU
USE MOD_GPU_Vars      ,ONLY: GPUInitialized
USE MOD_GPU_Interface ,ONLY: GPU_PushParticlesBatch
#endif /*PICLAS_USE_GPU*/

And the local flag variable:

#if PICLAS_USE_GPU
LOGICAL :: UseGPUPush
#endif /*PICLAS_USE_GPU*/

Step B â€” Eligibility check before the particle loop

The GPU push is only valid for the simple constant-dt case. It is disabled when any of the following features are active, because they either require per-particle dtVar or modify positions inside the loop in ways that the GPU kernel does not replicate: 

#if PICLAS_USE_GPU
UseGPUPush = GPUInitialized          .AND. &
             .NOT.UseVarTimeStep     .AND. &  ! GPU kernel uses constant dt
             .NOT.UseRotRefFrame     .AND. &  ! RotRef uses CalcPartPosInRotRef
             .NOT.UseGranularSpecies .AND. &  ! granular particles have their own push
             .NOT.DoSurfaceFlux      .AND. &  ! surface-flux particles use dtFracPush (random dt)
             .NOT.Symmetry%Axisymmetric        ! axisym calls CalcPartSymmetryPos on new positions
#endif /*PICLAS_USE_GPU*/

Step C â€” CPU loop: skip position update for GPU-eligible particles

The per-particle CPU loop still runs (it handles LastPartPos bookkeeping, surface-flux fractional-dt particles, and rotating-reference-frame particles). For GPU-eligible particles the position update is skipped with CONTINUE â€” the GPU kernel will do it below the loop: 

    IF(UseRotRefFrame) THEN
      LastPartVeloRotRef(1:3,iPart)=PartVeloRotRef(1:3,iPart)
      CALL CalcPartPosInRotRef(iPart, dtVar)
#if PICLAS_USE_GPU
    ELSE IF (UseGPUPush) THEN
      CONTINUE   ! position push offloaded to GPU batch call below the loop
#endif /*PICLAS_USE_GPU*/
    ELSE
      PartState(1:3,iPart) = PartState(1:3,iPart) + PartState(4:6,iPart) * dtVar
    END IF
LastPartPos is still set correctly LastPartPos(1:3,iPart) = PartState(1:3,iPart) executes inside the IF (.NOT.PDM%dtFracPush(iPart)) branch, which runs before the RotRef/GPU/CPU fork. So LastPartPos captures the position at the start of the time step regardless of which path does the actual push.

Step D â€” GPU batch call after the loop

#if PICLAS_USE_GPU
! GPU batch position push: upload PartState, run kernel, download results.
! Replaces the per-particle CPU push skipped with CONTINUE inside the loop.
IF (UseGPUPush) THEN
  CALL GPU_PushParticlesBatch(PartState, PDM%ParticleInside, PDM%ParticleVecLength, dt)
END IF
#endif /*PICLAS_USE_GPU*/

GPU_PushParticlesBatch converts PDM%ParticleInside (Fortran LOGICAL) to a C int mask in GPU_ActiveMask, then calls piclas_gpu_push_particles:

  1. H2D cudaMemcpy â€” uploads PartState(1:6, 1:nPart) and the mask to device
  2. Kernel launch â€” particle_push_kernel<<<ceil(nPart/256), 256>>> sets pos += vel * dt for every active particle (one thread each)
  3. D2H cudaMemcpy â€” downloads updated PartState back to host (implicit synchronisation point â€” no extra cudaDeviceSynchronize needed)
Memory layout Fortran stores PartState(1:6, iPart) column-major: all 6 components of particle iPart are contiguous in memory. The CUDA kernel maps iPart * 6 + 0..2 â†’ x,y,z and iPart * 6 + 3..5 â†’ vx,vy,vz, matching this layout exactly. No transposition is needed.

14.10 LSERK4 Time-Step Integration (PIC-EM, Step 2) NEW

Extends GPU acceleration to PP_TimeDiscMethod 2 (LSERK4-5, Carpenter 1994, 5 stages) and PP_TimeDiscMethod 6 (LSERK4-14, Niegemann 2012, 14 stages) â€” the two methods used for PIC-EM and combined DSMC+PIC-EM simulations. Both methods call the same TimeStepByLSERK() subroutine; only the compile-time RK coefficients and stage count differ.

Design guideline â€” Pt_temp residency (from Linux MPI source analysis)

Pt_temp follows the Linux MPI design: persistent host array, communicated with migrating particles

In the Linux MPI code (particle_mpi.f90), Pt_temp(1:6, iPart) is a persistent host-side array that lives for the entire timestep across all RK stages. When a particle crosses a domain boundary mid-sweep, the MPI layer packs Pt_temp(1:6) into the send buffer and unpacks it on the receiving rank â€” the accumulated derivative history travels with the particle so the RK integration can continue correctly without recomputation.

The GPU implementation mirrors this design exactly:

  • d_Pt_temp is allocated alongside d_PartState and initialized to zero (matching Pt_temp=0. in particle_init.f90).
  • Per stage: PartState and Pt_temp are uploaded (H2D) before the kernel and downloaded (D2H) after. The D2H of Pt_temp keeps the host copy in sync, which is the correct foundation for future MPI+GPU support â€” when a particle crosses a rank boundary after the kernel, the host Pt_temp already contains the updated value.
  • Pt (acceleration from Lorentz force), isActive, isPush, and isNewPart are also uploaded each stage.
  • For single-node builds (current target) there is no MPI particle exchange, so the Pt_temp D2H is cheap insurance rather than a correctness requirement.

15. superB Windows Port NEW

15.1 What superB Does

superB is a standalone magnetic-field pre-processor bundled with PICLas. It is a separate executable (not part of the main simulation loop) that:

  1. Reads a PICLas mesh file (HDF5).
  2. Evaluates coil magnetic fields via the Biot-Savart law (static) or Jefimenko equations (time-dependent), and permanent-magnet fields via magnetic scalar potential + gradient.
  3. Writes the resulting background B-field (BGField) to an HDF5 output file.
  4. That HDF5 file is referenced in the main PICLas parameter file so the field is loaded at runtime as a frozen background field.

Typical use case: compute the static field of a Helmholtz coil or a permanent-magnet ring, then run a combined DSMC+PIC-EM simulation that accelerates charged particles in that field.

15.2 Build Architecture

When PICLAS_PARTICLES=ON (the default for all maxwell/DSMC builds), CMake includes all src/posti/superB/*.f90 files in the main particlesF90 glob that feeds libpiclasshared. The physics modules (superB_coil, superB_permMag, superB_init, superB_main, superB_tools, superB_vars) are therefore compiled into libpiclas.dll as a side effect of any full PICLas build.

When PICLAS_BUILD_POSTI=ON the posti CMakeLists adds the superB standalone target: 

superB.exe â”€â”€linksâ”€â”€â–º superBlib   (OBJECT lib: superB.f90 entry point)
                    â–º libpiclasshared  (mesh, HDF5, all superB physics)
                    â–º ${linkedlibs}    (HDF5, OpenBLAS â€” WIN32 add_exec branch)
No extra DLLs needed for superB. Because the physics is already in libpiclas.dll, the superB.exe binary is tiny (~100 KB). It shares the same HDF5 and MPI infrastructure as piclas-win.exe and does not require any GPU libraries.

15.3 Windows Changes

The superB Fortran source is entirely cross-platform â€” no POSIX calls, no signal handlers, no Linux-specific I/O. Only two build-system changes were required:

Change 2 â€” Regex case fix in src/posti/superB/CMakeLists.txt

The PICLAS_PARTICLES=OFF branch used a lowercase regex to exclude the standalone entry point from the library source list: 

# Before (broken on case-sensitive and case-preserving filesystems):
LIST(FILTER superBnoparticlesF90 EXCLUDE REGEX ".*superb.f90$")

# After:
LIST(FILTER superBnoparticlesF90 EXCLUDE REGEX ".*[Ss]uper[Bb]\\.f90$")

The actual filename is superB.f90 (capital B). CMake REGEX is case-sensitive, so the original pattern silently failed to exclude the PROGRAM entry point from the library object list, which would cause a duplicate-symbol link error on Windows in no-particles builds. The fix uses a character class to match either case.

Scope: This code path is only active when PICLAS_PARTICLES=OFF (uncommon â€” requires explicitly disabling particle physics). All standard maxwell/DSMC/RK4 builds use PICLAS_PARTICLES=ON (the default) and are not affected.

Fix A2 — NIG_DVM BGKCollModel=4 crash — CRASH FIXED 2026-05-09

Suite affected: NIG_DVM (run_0004, DVM-Method=1 + BGKCollModel=4)

Root cause: src/equations/discrete_velocity/distfunc.f90:MaxwellDistributionCons computes alpha(1) = LOG(rho/…) − |u|²/… on entry. When the ED-DVM transport step produces a cell with infinite bulk velocity (|u| = ∞), alpha(1) = −∞. The Newton Jacobian J = <ψ exp(α·ψ)> becomes the zero matrix (EXP(−∞)=0), causing LAPACK DGESV to fail with “singular matrix”.

Models 1–3 (ESBGK, Shakhov, Maxwell) handle infinite velocity gracefully because their distribution formula contains EXP(−|c|²/2RT) where c = v − u; when u=∞, |c|=∞ and EXP(−∞)=0 → zero output. The conservative Maxwell Newton solver has no such implicit guard.

Fix A2 applied (distfunc.f90, MaxwellDistributionCons): guard for infinite or non-physical bulk velocity on entry:

! non-physical MacroValues: return empty distribution (matches MaxwellDistribution behaviour)
IF (rho.LE.0.0 .OR. Temp.LE.0.0 .OR. DOTPRODUCT(uVelo).GE.HUGE(rho)) THEN
  fMaxwell = 0.0
  RETURN
END IF

Fix A2b — NIG_DVM BGKCollModel=4 analyze error — FIXED 2026-05-09

Suite affected: NIG_DVM run_0004 (ED-DVM + BGKCollModel=4), analyze error L2 x-velocity 42 vs threshold 2.

Root cause: After Fix A2, cells in the ED-DVM transport with large-but-finite bulk velocity (e.g. 105 m/s from near-zero-density cells where uVelo = rhoU/dens with tiny dens just above the 0.1 skip threshold) bypass the DOTPRODUCT≥HUGE guard. In the Newton loop, EXP(alpha·ψ) overflows where alpha(2) = uVelo/(RST) and ψ(2) = v_x is a large Gauss-Hermite node; J = ∞ → DGESV info=1 → wrong output instead of zero.

Fix A2b applied: After alpha initialization, compute the maximum linear-exponent magnitude across all velocity dimensions and grid nodes; if it exceeds 700 (≈ ln(HUGE_double)), return zero distribution:

! Added to local variables:
INTEGER :: idim
REAL    :: maxAlphaPsi

! After alpha(2+DVMDim) = -1/R_S/Temp, before countz=0:
maxAlphaPsi = 0.0
DO idim = 1, DVMDim
  maxAlphaPsi = MAX(maxAlphaPsi, ABS(alpha(1+idim)) * MAXVAL(ABS(DVMSpecData(iSpec)%Velos(:,idim))))
END DO
IF (.NOT.(maxAlphaPsi .LT. 700.0)) THEN  ! catches Inf and NaN
  fMaxwell = 0.0
  RETURN
END IF

The idiom .NOT.(x .LT. 700.0) correctly handles NaN (NaN<700 is .FALSE.; .NOT..FALSE. = .TRUE. → guard triggers).

The shell script that drives all NIG_ suites used a pipeline that made every suite appear to pass:

    echo “  >>> $SUITE: PASSED”
    pass=$(( pass + 1 ))
else

§16.14 WEK_Reservoir Analyze Failures — MS-MPI MPI_REDUCE Fix FIXED 2026-05-20

Running the WEK_Reservoir regression suite with 6 MPI ranks produced 48 analyze failures (out of 80 total examples). All failures traced to the same root cause in MS-MPI.

Root Cause

MS-MPI has a well-known bug with MPI_REDUCE(MPI_IN_PLACE, buf, …):

  • MPI=1: result is 0 (root's contribution entirely zeroed).
  • MPI=N: result is the sum of ranks 1…N−1; root's value is dropped.

On Linux the in-place form is legal (root provides both input and output buffer). MS-MPI incorrectly treats root's buffer as an empty receive buffer instead of reading it as the send value before reducing into it.

The pattern appears throughout src/particles/analyze/ where global reduce-sums are accumulated into the same array used as both operand and result:

! Buggy pattern on MS-MPI:
CALL MPI_REDUCE(MPI_IN_PLACE, arr, n, MPI_DOUBLE_PRECISION, MPI_SUM, 0, MPI_COMM_WORLD, iError)

! Fixed pattern:
tmpArr = arr
CALL MPI_REDUCE(tmpArr, arr, n, MPI_DOUBLE_PRECISION, MPI_SUM, 0, MPI_COMM_WORLD, iError)

The failure signature was: every failing column showed a ratio close to (N−1)/N = 5/6 compared to the reference, confirming root's contribution was missing.

Fixes Applied

Seven MPI_REDUCE(MPI_IN_PLACE, …) calls were replaced with explicit temporary buffers across two files:

Titan Chemistry Platform Difference

The final remaining failure was CHEM_EQUI_Titan_Chemistry run 2 (Particles-DSMC-ElectronicModel=4, BGK-LT relaxation) with MPI=6. The column deviations ranged from 0.5× to 3.7× — not the uniform 5/6 ratio that would indicate an MPI_IN_PLACE bug — ruling out a code defect.

Root cause: the stochastic DSMC equilibrium chemistry simulation produces a platform-dependent random-number trajectory. The BGK-LT electronic model (ElecModel=4) consumes more random numbers per collision step than the Liechty model (ElecModel=1), so by t=5 μs the Windows and Linux simulations have explored entirely different regions of phase space. §16.17 OS Freeze Prevention — Memory-Based Particle Cap FIXED 2026-05-21

Incident

After the §16.16 rebuild, running NIG_DSMC still produced a complete OS freeze requiring a hard reset. An external watchdog (6 GB RSS cap, 240 s timeout) did not help — the system froze so hard that even the clock stopped.

Why a Polling Watchdog Cannot Help

A particle-insertion runaway commits memory faster than any polling watchdog samples it. Once the Windows commit charge exceeds physical RAM, the kernel thrashes the pagefile and no user-mode process — including the watchdog — gets scheduled. The stopped clock is the signature of this paging death. Prevention therefore has to be in-process and synchronous, not external and polled.

Root Cause

PICLas already contains a clean abort for runaway growth — IncreaseMaxParticleNumber() in src/particles/particle_tools.f90:2504 calls ABORT('More Particles needed than allowed in PDM%maxAllowedParticleNumber'). That guard is bounded by Part-maxParticleNumber, which defaulted to HUGE(integer) ≈ 2.1 billion (src/particles/particle_init.f90:540–541). The safety net existed but was effectively disabled: a runaway grows the particle arrays toward 2.1 billion entries (hundreds of GB) and exhausts RAM long before the abort can fire. This is independent of which MPI_IN_PLACE divisor bug triggers the runaway — the cap is the path-independent last line of defence.

Fix Applied

The default for Part-maxParticleNumber is now derived from installed physical RAM, so the existing abort fires at a few GB instead of at hundreds of GB. The value is still fully overridable from parameter.ini for genuine large production runs.

§16.18 GPU Managed-VRAM OOM — Diagnosis & Memory-Model Rework Plan DIAGNOSED 2026-05-22

Incident & Log Evidence

A full NIG_ regression run (nig_all_results.log, 2026-05-22) did not abort globally, but multiple GPU runs crashed with CUDA out-of-memory. The failures cluster in the two suites that sweep increasing MPI rank counts on a single GPU:

Every failure carries the same signature — the first cudaMallocManaged of d_PartState at gpu_memory.cu:73:

[GPU] CUDA error at ...src\gpu\gpu_memory.cu:73 — out of memory
Smoking gun — it is NOT a particle runaway
Log line 40447, at MPI = 10:
[GPU] Managed buffers: 100001 particles, 8.0 MB (VRAM 11245.0 / 12287.5 MB)
The buffer being allocated is 8 MB for 100 k particles — trivial. But VRAM is already at 11245 / 12287 MB (91 %) before that allocation. This is not the §16.15/§16.16 adaptive-BC runaway (already fixed) and not the §16.17 particle cap. The VRAM is pre-consumed by something else entirely.

Root Cause — N MPI Ranks on One Physical GPU

gpu_init.cu:36 unconditionally calls cudaSetDevice(0) on every MPI rank. All ranks therefore pile onto the single 12 GB card. Each rank creates its own CUDA context + runtime + managed-memory pool. Under the Windows WDDM driver this is on the order of ~1 GB of VRAM per rank before any particle data is touched. With ~10 ranks co-resident, 10 × ~1.1 GB ≈ 11 GB, so the next rank’s 8 MB allocation tips the card over.

The decisive consequence: the OOM threshold tracks MPI rank count, not particle count. The heavier DSMC case dies at ~9 ranks; the lighter maxwell-RK4 case survives to ~17. The reggie suites mortar_exchange_procs and 3D_periodic_CVWM deliberately sweep MPI = 1 … 25, so they will always hit this ceiling on a single-GPU desktop.

Why Managed VRAM Fails on Windows (the regression from this change)

The switch from unmanaged (cudaMalloc) to managed (cudaMallocManaged) VRAM is what turned graceful behaviour into hard crashes. The design comment at gpu_memory.cu:52–56 states managed memory will “transparently page to system RAM when VRAM pressure is high”, preventing hard OOM. That premise is false on consumer Windows GPUs.

  • Unified-Memory oversubscription (allocating beyond physical VRAM and paging to host RAM on demand) is supported on Linux, and on Windows only under the TCC driver.
  • Consumer GeForce / RTX cards on Windows run the WDDM driver, where managed allocations must remain VRAM-resident and simply fail with cudaErrorMemoryAllocation when VRAM is exhausted — exactly the crash observed.

Worse, the budget helper piclas_gpu_query_max_safe() (gpu_memory.cu:140–175) computes budget = vram_free + sys_use (line 164) — it counts system RAM that cannot actually be used under WDDM, and it uses the whole free-VRAM figure without dividing by the number of ranks sharing the device. So even the “safe cap” structurally over-promises and never prevents the OOM.

Immediate mitigations (no rebuild)
  1. Regression testing → use a CPU-only binary (PICLAS_USE_GPU=OFF). The GPU only accelerates the position push; DSMC surface-flux already runs on CPU (§16.15), and these cases hold ~100 k particles (8 MB) — the GPU gives no measurable speedup here and is the sole source of these crashes. A CPU build also removes the Bug B GPU-vs-CPU floating-point reference mismatches (§16.9/§16.10).
  2. If the GPU binary must run: cap ranks-per-GPU with reggie’s -l (e.g. -l 4, §16.4), or skip the MPI ≥ 9 variants. Do not sweep MPI = 9 … 25 on one 12 GB card.

Memory-Model Rework Plan (long-term: safe and fast)

The current model is unsafe (managed oversubscription is a no-op on WDDM) and slow (full PartState H2D+D2H every RK stage — gpu_memory.cu:192–205, 236–267 — for a kernel that only does pos += vel*dt, so PCIe transfer dominates and the GPU yields ~no acceleration). The rework targets both. Phases are ordered by impact-to-effort; each is independently shippable.

Phase 1 — Rank→GPU binding & VRAM partitioning (fixes the crash)

  • Replace the blind cudaSetDevice(0) (gpu_init.cu:36) with a binding based on the node-local MPI rank: split MPI_COMM_WORLD with MPI_COMM_TYPE_SHARED, take the local rank/size, then cudaSetDevice(localRank % devCount). On multi-GPU nodes this spreads ranks across cards; on a single-GPU desktop it makes ranksPerGPU = localSize explicit and known.
  • Pass ranksPerGPU into the budget query and partition VRAM: perRankBudget = (vram_free − contextReserve) / ranksPerGPU. Drop system RAM from the budget entirely (it is unusable under WDDM). This makes the “safe cap” real.
  • Recommend / document NVIDIA MPS for many-rank single-GPU runs: a single shared CUDA context removes the per-rank ~1 GB context overhead and is the proper way to oversubscribe one GPU with many MPI ranks.

Phase 2 — Unmanaged buffers + deterministic batching (safe sizing)

  • Switch the six device buffers from cudaMallocManaged to plain cudaMalloc (gpu_memory.cu:72–85). Size them to min(perRankBudget, maxParticleNumber) — a fixed buffer that is guaranteed to fit.
  • When a rank’s live particle count exceeds its buffer capacity, process the push in chunks (a host-side loop over [offset, offset+chunk) slices, each ≤ buffer size), instead of growing toward VRAM exhaustion. This is the “split into batches that fit GPU memory” approach — correct, but note it only addresses the single-rank-too-many-particles axis; Phase 1 is what addresses the many-ranks axis seen in this log.
  • Replace the free-then-realloc growth path (gpu_interface.f90:177–195, 241–259) with a fixed allocation + chunked streaming, eliminating fragmentation and transient double-allocation.
  • Graceful CPU fallback: if even one chunk cannot be allocated, fall back to the CPU push for that step rather than exit(EXIT_FAILURE) (gpu_memory.cu:25).

Phase 3 — GPU-resident state & transfer minimisation (the speed win)

  • Keep PartState (and Pt_temp) resident on the device across all RK stages and across timesteps. Upload once when particles change on the host (insertion, emission, MPI migration), download only when the host needs them (output, tracking, analyze) — not every stage. This removes the per-stage PCIe ping-pong that currently makes the GPU path slower than CPU for small cases.
  • Use pinned (page-locked) host memory for the staging arrays and asynchronous cudaMemcpyAsync + streams so the remaining transfers overlap kernel execution and MPI communication.
  • For MPI runs, transfer only the delta — the particles that migrated across the halo — rather than the full array, aligning the GPU data motion with the existing particle MPI exchange.
  • Move more of the per-step particle work (interpolation, Lorentz force, deposition) onto the device so the resident data is actually exploited; a kernel that only does pos += vel*dt can never amortise a full-array round trip.

Phase 4 — Validation & guard rails

  • Keep the §16.17 RAM-based particle cap as the path-independent last line of defence.
  • Add a startup banner line reporting device id, ranksPerGPU, perRankBudget, chunk size, resident vs. streamed mode so future OOMs are self-diagnosing.
  • Regression-validate with a CPU-only reference (Bug B): once Windows GPU references are regenerated, the GPU and CPU binaries should agree within tolerance.
Summary of the recommendation Your instinct — drop managed VRAM, use a fixed buffer that fits and batch — is correct for Windows/WDDM and is captured in Phase 2. But the crash in this log is rank-count driven, so Phase 1 (rank→GPU binding + VRAM partitioning, or MPS) is the part that actually stops it. Phase 3 (GPU-resident state) is where the real speed-up comes from. For the immediate need, run regression on a CPU-only binary; pursue Phases 1–3 for a long-term stable, fast GPU build.

The mortar false failure comes entirely from the analyze step comparing PartInt (per-element particle counts) by exact h5diff against a Linux-generated reference. PartInt is the worst possible quantity to compare bit-for-bit: it is exactly the FP-tie-break-sensitive integer that flips for on-face particles. The fix (consistent with the DSMC/Monte-Carlo philosophy of validating statistics and invariants, not bits) is to drop the PartInt diff and assert the physical invariants that the case actually guarantees.

Why this is the right fix, not a tolerance hack. Note that energy conservation already subsumes a particle-count check here: losing even one of the 2000 particles drops the kinetic energy by ~1/2000 (rel. ~5×10−4), far above the 10−8 tolerance — so an explicit CalcNumSpec column is optional. If a bit-reproducible per-element check is ever needed, it must use a Windows-generated PartInt reference and is only valid for same-platform, same-compiler runs.

§16.21.5 RotPeriodicBCMultiInterPlane — SIGFPE from a 1-leader MPI_IN_PLACE reduce

NIG_DSMC/RotPeriodicBCMultiInterPlane SIGFPE'd deterministically at MPI=2 and 7, right after the first load balance — which previously got it mis-filed as Bug G. gdb (deterministic crash → gdb catches it) pinned it to surfacemodel_analyze.f90 GetGroupInfo: SampState(4,iGroup) / (TimeSampleTemp * SurfaceGroup%Area(iGroup)) with Area=0.

Every rank computes the full single-node group area (the area loop covers all nComputeNodeSurfTotalSides). The follow-up MPI_REDUCE(MPI_IN_PLACE, …, MPI_COMM_LEADERS_SHARED) — intended only to combine areas across multiple compute nodes — runs on a 1-member communicator on a single node, where MS-MPI zeroes the MPI_IN_PLACE buffer (the §13.1 bug), setting Area=0. The divide only fires once a group has wall hits (Counter>0), hence the “after load balance” timing (really: after particles reach the rotating wall). Fix: guard the reduce with IF(nLeaderGroupProcs.GT.1) (root already holds the full area). Verified MPI=2,7; reggie surface comparisons (GroupID, Group_Symmetry_fac, InterPlanePosition) match references.

§16.26 GPU Push — Surface Flux + Axisymmetric DSMC Support

As of piclas-win 1.0 the CUDA particle-position push supports two DSMC configurations that were previously forced onto the CPU path: constant/adaptive surface-flux emission and 2D axisymmetric runs. This directly benefits cases such as the silver-vapor nozzle (Symmetry2DAxisymmetric = T with an adaptive surface-flux inlet), whose position push now runs on the GPU.

§16.26.1 Previous limitation

The UseGPUPush guard in TimeStep_DSMC disqualified any run with DoSurfaceFlux = T or Symmetry%Axisymmetric = T. The kernel only computed pos += vel * dt, so it handled neither the fractional time step of freshly inserted surface-flux particles nor the axisymmetric rotation of the radial coordinates; enabling the GPU path for those cases would have produced wrong positions, so they silently fell back to the CPU.

§16.26.2 Patch

  • particle_push_kernel now takes dtFracPush[nPart], dtFracRand[nPart] and symmetryOrder. Freshly inserted surface-flux particles have their dt scaled by the host-supplied uniform random number; for symmetryOrder = 2 or 3 the kernel rotates the (y,z) or (x,z) components per CalcPartSymmetryPos. The on-axis |r| < eps case leaves the velocity unchanged.
  • The random numbers are drawn host-side in iPart order (in GPU_PushParticlesBatch, gpu_interface.f90) so the global RANDOM_NUMBER sequence stays bit-identical to the CPU per-particle loop — GPU and CPU runs remain reproducible.
  • New device buffers d_dtFracPush / d_dtFracRand are allocated alongside d_PartState, and the UseGPUPush guard is relaxed to admit surface-flux and axisymmetric runs.
Still CPU-only: ambipolar diffusion (DSMC%DoAmbipolarDiff), variable / species-specific time step, rotational reference frame, and granular species — the kernel does not yet handle their extra per-particle work, so those runs continue to use the CPU push.
