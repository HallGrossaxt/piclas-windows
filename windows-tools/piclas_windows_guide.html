Overview

PICLas is a Particle-in-Cell / Direct Simulation Monte Carlo (PIC-DSMC) plasma simulation code written in Fortran/C. It was originally developed for Linux/HPC environments. This guide documents all steps taken to build and run PICLas on Windows 11 using the MSYS2 UCRT64 environment, including source code patches, build system configuration, and runtime fixes.

Key strategy PICLas is compiled natively on Windows using GCC/GFortran provided by MSYS2 (not Cygwin, not WSL). The resulting piclas.exe is a real Windows executable that runs without MSYS2 at runtime, provided the required DLLs are bundled (see Distribution).

4. Source Code Changes

PICLas requires four modifications to build and run correctly on Windows. These changes are backward-compatible: on Linux/macOS the new C file compiles to a no-op stub and behaviour is unchanged.

4.1 Windows Wildcard Expansion — glob_windows.c NEW FILE

On Linux, the shell expands wildcards like plasma_wave_State_*.h5 before passing them to the program. Windows shells (CMD, PowerShell) do not — the literal string * reaches the program unchanged. This new file implements wildcard expansion using the Win32 API so that piclas2vtk parameter.ini State_*.h5 works from any Windows terminal.

File: piclas-master/src/globals/glob_windows.c 
// Win32 glob expansion — called from Fortran via ISO_C_BINDING
// On non-Windows: stub always returns 0 (shell already expands globs)
#ifdef _WIN32
#include <windows.h>
#include <string.h>
#include <stdlib.h>

static int cmp_str(const void *a, const void *b) {
    return strcmp(*(const char **)a, *(const char **)b);
}

int glob_expand_c(const char *pattern, char *results, int results_len) {
    WIN32_FIND_DATAA fd;
    HANDLE h;
    char *ptrs[4096];
    int count = 0;
    int pos = 0;

    h = FindFirstFileA(pattern, &fd);
    if (h == INVALID_HANDLE_VALUE) return 0;

    /* Determine directory prefix from pattern */
    char dir[MAX_PATH] = "";
    const char *slash = strrchr(pattern, '/');
    const char *bslash = strrchr(pattern, '\\');
    const char *last = slash > bslash ? slash : bslash;
    if (last) { int n = (int)(last - pattern) + 1; memcpy(dir, pattern, n); dir[n] = 0; }

    do {
        if (count >= 4096) break;
        char *entry = malloc(MAX_PATH);
        snprintf(entry, MAX_PATH, "%s%s", dir, fd.cFileName);
        ptrs[count++] = entry;
    } while (FindNextFileA(h, &fd));
    FindClose(h);

    qsort(ptrs, count, sizeof(char *), cmp_str);

    for (int i = 0; i < count; i++) {
        int len = strlen(ptrs[i]);
        if (pos + len + 1 >= results_len) { free(ptrs[i]); break; }
        memcpy(results + pos, ptrs[i], len + 1);
        pos += len + 1;
        free(ptrs[i]);
    }
    return count;
}
#else
int glob_expand_c(const char *pattern, char *results, int results_len) {
    (void)pattern; (void)results; (void)results_len;
    return 0;  /* Linux/macOS: shell handles glob expansion */
}
#endif

4.2 Fortran Command-Line Interface — commandlinearguments.f90 MODIFIED

Add an ISO_C_BINDING interface for glob_expand_c and call it for any argument containing * or ?.

File: piclas-master/src/globals/commandlinearguments.f90

Add at the top of the module, inside the relevant subroutine:

USE ISO_C_BINDING, ONLY: C_INT, C_CHAR, C_NULL_CHAR

INTERFACE
  FUNCTION glob_expand_c(pattern, results, results_len) BIND(C, NAME='glob_expand_c')
    USE ISO_C_BINDING, ONLY: C_INT, C_CHAR
    CHARACTER(KIND=C_CHAR), INTENT(IN)        :: pattern(*)
    CHARACTER(KIND=C_CHAR), INTENT(OUT)       :: results(*)
    INTEGER(KIND=C_INT),    VALUE, INTENT(IN) :: results_len
    INTEGER(KIND=C_INT)                       :: glob_expand_c
  END FUNCTION glob_expand_c
END INTERFACE

Then wrap each argument that contains * or ? through glob_expand_c before storing it.

5. Build Fixes

5.1 TEMP/TMP environment variables

On some Windows systems, the TEMP and TMP variables point to C:\Windows\Temp (a system directory). GCC cannot write temporary files there and aborts with "Cannot create temporary file in C:\WINDOWS\: Permission denied".

Set them to the user temp directory before building. Add these lines to any build script:

export TEMP='/c/Users/YOUR_USERNAME/AppData/Local/Temp'
export TMP=$TEMP

Or in a .bat file, use /tmp which MSYS2 maps to C:\msys64\tmp:

"%BASH%" -c "export TEMP=/tmp && export TMP=/tmp && cmake --build ..."

5.2 Missing Fortran module include directory

GFortran warns about C:/msys64/ucrt64/mod/static not existing. The directory is referenced by the HDF5 Fortran CMake configuration for static library modules. The warning is harmless but can be silenced by creating the empty directory: 

mkdir -p /c/msys64/ucrt64/mod/static

5.3 PETSc pkg-config wrapper FIX

MSYS2 installs PETSc with variant-specific .pc files (petsc-dso.pc, petsc-cso.pc, etc.) but not a plain petsc.pc. PICLas's FindPETSc.cmake uses pkg-config to search for the package name petsc and, when it fails to find it, falls back to downloading and compiling PETSc from source — which fails on Windows because the PETSc configure script uses Unix environment-variable syntax incompatible with cmd.exe.

The fix is to create a plain petsc.pc wrapper pointing to the dso variant (double precision, serial, optimized): 

# Create file: C:\msys64\ucrt64\lib\pkgconfig\petsc.pc
prefix=/c/msys64/ucrt64
exec_prefix=${prefix}
libdir=${exec_prefix}/lib/petsc/dso
includedir=${exec_prefix}/include/petsc

Name: PETSc
Description: Sequential double precision PETSc build
Version: 3.24.5
Requires.private: hwloc openblas
Cflags: -I${includedir}/dso -I${includedir}
Libs: -L${libdir} -lpetsc
Libs.private: -L${libdir} -lpetsc -lgfortran -lquadmath

Create this file from the MSYS2 shell:

cat > /c/msys64/ucrt64/lib/pkgconfig/petsc.pc << 'EOF'
prefix=/c/msys64/ucrt64
exec_prefix=${prefix}
libdir=${exec_prefix}/lib/petsc/dso
includedir=${exec_prefix}/include/petsc

Name: PETSc
Description: Sequential double precision PETSc build
Version: 3.24.5
Requires.private: hwloc openblas
Cflags: -I${includedir}/dso -I${includedir}
Libs: -L${libdir} -lpetsc
Libs.private: -L${libdir} -lpetsc -lgfortran -lquadmath
EOF

Verify it works:

pkg-config --modversion petsc
# Expected output: 3.24.5

When building with PETSc, also pass these cmake flags to ensure the system library is used rather than triggering a source build:

-DLIBS_BUILD_PETSC=OFF \
-DPETSC_LINK_LIBRARIES=C:/msys64/ucrt64/lib/petsc/dso/libpetsc.dll.a

And set the environment variable so FindPETSc.cmake finds the pkgconfig directory:

export PETSC_DIR=/c/msys64/ucrt64

5.4 Parallel build race condition (unit tests)

When Ninja builds in parallel, unit test source files can start compiling before the Fortran .mod files in libpiclasstaticF90 have been generated, causing "Fatal Error: Cannot open module file 'mod_globals.mod'".

Workaround: build with --target piclas to skip unit tests, or set PICLAS_UNITTESTS=OFF. The CMake dependency fix in unitTests/CMakeLists.txt (Section 4.3) also mitigates this.

6. MPI Build (MS-MPI)

PICLas can be built with MPI support on Windows using Microsoft MPI (MS-MPI v10.1) provided by MSYS2. This enables multi-process parallel simulations via mpiexec -n N piclas.exe. Note that MSYS2's HDF5 is built without parallel MPI support, so file I/O remains serial even in MPI runs — this is the standard behaviour for most desktop/workstation use cases.

6.1 Install MS-MPI Package

Install the MSYS2 MS-MPI wrapper package (provides mpicc, mpifort, import library libmsmpi.dll.a, and header mpi.h):

pacman -S --noconfirm mingw-w64-ucrt-x86_64-msmpi
MS-MPI runtime The MSYS2 package provides the import library for linking. For running MPI programs you also need the MS-MPI runtime DLL (msmpi.dll). This is provided by Microsoft MPI v10 and is typically already installed as part of the MSYS2 package or can be installed separately.

6.2 Source Code and CMake Fixes

Eight fixes are needed to get the MPI build working correctly on Windows. These are all already applied in the patched source tree.

Fix A — mpi_f08.o linker ordering (cmake/SetLibraries.cmake)

MS-MPI v10 provides only the legacy MODULE MPI interface. PICLas ships a compatibility wrapper cmake/mpi_f08.f90 that provides the modern typed MPI_F08 interface, compiled into mpi_f08.o. GNU ld scans archives left-to-right: the object file must appear before libmsmpi.dll.a so that its undefined references (mpi_wtick_ etc.) are known when the import library is scanned. Changed from LIST(APPEND…) to LIST(INSERT … 0 …): 

# mpi_f08.o must precede libmsmpi.dll.a in the link command
IF(EXISTS "${_mpi_f08_dir}/mpi_f08.o")
  LIST(INSERT linkedlibs 0 "${_mpi_f08_dir}/mpi_f08.o")
ENDIF()

Fix B — HDF5 parallel detection false positive (cmake/SetLibraries.cmake)

MSYS2 HDF5 is built without parallel MPI support (HDF5_PROVIDES_PARALLEL=OFF in hdf5-config.cmake). The original detection used grep mpio_f which matched the symbol h5fd_mpio_f_ (an MPIO file-driver constant that is always present), causing a false HDF5_IS_PARALLEL=TRUE. Changed to search for the specific function that only exists in parallel-capable builds: 

# Search specifically for h5pset_fapl_mpio_f to avoid false positives from h5fd_mpio_f
ELSEIF(WIN32)
  EXECUTE_PROCESS(COMMAND nm -g ${HDF5_Fortran_LIBRARY_hdf5_fortran}
                  COMMAND grep h5pset_fapl_mpio_f
                  OUTPUT_VARIABLE HDF5_USES_MPIF ...)

Fix C — USE_MPI_HDF5 compile flag + source guards (cmake/SetLibraries.cmake, 3 HDF5 source files)

Added a new preprocessor flag that is 1 only when both MPI is enabled AND HDF5 was built with parallel support. This guards the 5 places in the source that call H5PSET_FAPL_MPIO_F or H5PSET_DXPL_MPIO_F (which do not exist in the serial MSYS2 HDF5 library): 

# cmake/SetLibraries.cmake — after HDF5 detection:
IF(LIBS_USE_MPI AND HDF5_IS_PARALLEL)
  ADD_COMPILE_DEFINITIONS(USE_MPI_HDF5=1)
ELSE()
  ADD_COMPILE_DEFINITIONS(USE_MPI_HDF5=0)
ENDIF()

In the three HDF5 source files, the parallel I/O calls are wrapped with #if USE_MPI_HDF5 instead of the former #if USE_MPI:

! io_hdf5.f90 / hdf5_input.f90 / hdf5_output.f90
#if USE_MPI_HDF5
  CALL H5PSET_FAPL_MPIO_F(Plist_File_ID, communicatorOpt%mpi_val, MPIInfo%mpi_val, iError)
#endif /*USE_MPI_HDF5*/

Fix D — Full path to powershell.exe in userblock step (CMakeLists.txt)

The userblock pre-build step invokes generateuserblock.ps1 via powershell. During the Ninja build, Windows PowerShell is not in the effective PATH. Fixed by using the full path: 

ADD_CUSTOM_COMMAND(TARGET libpiclasstatic PRE_BUILD COMMAND
  "C:/Windows/System32/WindowsPowerShell/v1.0/powershell.exe"
  -ExecutionPolicy Bypass -NonInteractive -File
  "${CMAKE_CURRENT_SOURCE_DIR}/tools/userblock/generateuserblock.ps1" ...)

Fix E — Link MPI libs directly to executables (src/CMakeLists.txt, unitTests/CMakeLists.txt)

On Windows, symbols from a DLL's transitive dependencies are not re-exported through the DLL's import stub (liblibpiclas.dll.a). So mpi_finalize_ and other MPI symbols that libpiclas.dll uses internally are not visible to piclas.exe when it links only against the import stub. Fixed by linking ${linkedlibs} (which contains mpi_f08.o and libmsmpi.dll.a) directly to both the main executable and unit test executables on Windows: 

# src/CMakeLists.txt — add_exec function
FUNCTION(add_exec target)
  ADD_EXECUTABLE(       ${target} ${ARGN})
  TARGET_LINK_LIBRARIES(${target} libpiclasshared)
  IF(WIN32)
    TARGET_LINK_LIBRARIES(${target} ${linkedlibs})
    ...

# unitTests/CMakeLists.txt — add_unit_test function
IF(WIN32)
  TARGET_LINK_LIBRARIES(${target} ${linkedlibs})
ENDIF()

Fix F — Userblock / HDF5 superblock corruption (src/io_hdf5/io_hdf5.f90)

When copy_userblock copies the Fortran-generated userblock to the front of an HDF5 file, it must reserve space before the HDF5 superblock. Without this fix the userblock overwrites the superblock, corrupting the file. The fix calls H5PSET_USERBLOCK_F on the property list before H5FCREATE_F, and passes creation_prp=Plist_File_ID so HDF5 reserves the correctly rounded-up 512-byte-aligned block: 

! io_hdf5.f90 — OpenDataFile, inside the MPIRoot / create-new-file path
IF(userblockSize_loc.GT.0)THEN
  tmp = userblockSize_512
  DO WHILE(tmp.LT.userblockSize_loc)
    tmp = tmp * 2
  END DO
  CALL H5PSET_USERBLOCK_F(Plist_File_ID, tmp, iError)
END IF
CALL H5FCREATE_F(TRIM(FileString), H5F_ACC_TRUNC_F, File_ID, iError, creation_prp=Plist_File_ID)

Fix G — Sequential HDF5 gather-write pattern (src/io_hdf5/hdf5_output.f90)

With USE_MPI=1, USE_MPI_HDF5=0 (MPI compute, serial HDF5), only MPIRoot opens the HDF5 file — non-root ranks have File_ID=0. Calling WriteArrayToHDF5 directly on all ranks would cause H5DCREATE_F(File_ID=0) to fail on ranks 1…N with "Dataset could not be created".

GatheredWriteArray and DistributedWriteArray handle this correctly: each non-root rank sends its local slice to MPIRoot via MPI_GATHERV, and only MPIRoot calls the HDF5 write. The key guard in these routines is: 

! hdf5_output.f90 — GatheredWriteArray
#if USE_MPI && !USE_MPI_HDF5
  ! Gather data from all ranks to MPIRoot, then only MPIRoot writes
  CALL MPI_GATHERV(SendBuf, SendCount, MPI_DOUBLE_PRECISION, &
                   RecvBuf, RecvCounts, Displs, MPI_DOUBLE_PRECISION, &
                   0, MPI_COMM_PICLAS, iError)
  IF(MPIRoot) CALL WriteArrayToHDF5(...)
#else
  CALL WriteArrayToHDF5(...)
#endif

Fix H — WriteDSMCToHDF5 uses GatheredWriteArray (src/particles/dsmc/dsmc_analyze.f90)

WriteDSMCToHDF5 previously opened the DSMCState file with OpenDataFile(single=.FALSE., communicatorOpt=MPI_COMM_PICLAS) and then called WriteArrayToHDF5 directly on all ranks. This bypassed the gather-write pattern and caused "ERROR in WriteArrayToHDF5: Dataset ElemData could not be created" on ranks 1–3.

Fix: removed the OpenDataFile / CloseDataFile wrapper and replaced the direct WriteArrayToHDF5 calls with GatheredWriteArray. Added GatheredWriteArray to the USE MOD_HDF5_Output list: 

! dsmc_analyze.f90 — WriteDSMCToHDF5 (replaces OpenDataFile + WriteArrayToHDF5 + CloseDataFile)
USE MOD_HDF5_Output ,ONLY: WriteAttributeToHDF5, WriteHDF5Header, WriteArrayToHDF5, GatheredWriteArray

ASSOCIATE (&
  nVarX        => INT(nVar+nVar_quality+nVar_HeatPress+nVarMPF,IK) ,&
  PP_nElems    => INT(PP_nElems,IK)                                 ,&
  offsetElem   => INT(offsetElem,IK)                                ,&
  nGlobalElems => INT(nGlobalElems,IK)                           )
  CALL GatheredWriteArray(FileName,create=.FALSE.,DataSetName='ElemData', rank=2,  &
                          nValGlobal=(/nVarX,nGlobalElems/),                        &
                          nVal      =(/nVarX,PP_nElems/),                           &
                          offset    =(/0_IK ,offsetElem/),                          &
                          collective=.false., RealArray=DSMC_MacroVal(:,:))
  IF(SampleElecExcitation) THEN
    CALL GatheredWriteArray(FileName,create=.FALSE.,DataSetName='ExcitationData', rank=2, &
                            nValGlobal=(/nVarExci,nGlobalElems/),                          &
                            nVal      =(/nVarExci,PP_nElems/),                             &
                            offset    =(/0_IK    ,offsetElem/),                            &
                            collective=.false., RealArray=MacroElecExcitation(:,:))
  END IF
END ASSOCIATE

6.4 Running with MPI

# Single process (same as non-MPI)
./build-ucrt64-mpi/bin/piclas.exe parameter.ini

# Multi-process (requires msmpi.dll in PATH)
mpiexec -n 4 ./build-ucrt64-mpi/bin/piclas.exe parameter.ini
Required: disable HDF5 file locking before mpiexec On Windows, HDF5's file-locking mechanism conflicts with how Windows manages file handles across multiple processes. Set this environment variable before running any MPI simulation, otherwise PICLas will abort during the first HDF5 write: 
# MSYS2 / bash
export HDF5_USE_FILE_LOCKING=FALSE
mpiexec -n 4 bin/piclas.exe parameter.ini DSMC.ini

HDF5 file I/O is serial — gather-write pattern Since MSYS2's HDF5 has no parallel MPI support (USE_MPI_HDF5=0), only MPIRoot (rank 0) opens HDF5 files. Non-root ranks have File_ID=0 and must not call any HDF5 write routines directly. PICLas handles this transparently via the gather-write pattern:
  • Each rank sends its local data slice to MPIRoot via MPI_GATHERV
  • MPIRoot assembles the full array and writes it to HDF5
  • This is done inside GatheredWriteArray and DistributedWriteArray
Any call to WriteArrayToHDF5 directly on all ranks (bypassing the gather-write) will fail on ranks 1–N with "Dataset could not be created". See Fix G and Fix H above.

13. Runtime Bugs (Landau Damping PIC-Poisson Tutorial)

Three Windows-specific runtime bugs were found and fixed while verifying the pic-poisson-landau-damping tutorial with mpiexec -n 4. All three are caused by differences between MS-MPI and the Linux MPI implementations (OpenMPI/MPICH) that PIClas was primarily developed and tested against. None affect single-process runs or Linux.

13.1 MS-MPI MPI_ALLGATHERV(MPI_IN_PLACE) corrupts SideInfo_Shared

Symptom

FATAL ERROR: GlobalSideID not found for Elem …

Occurs immediately during mesh initialisation on every MPI rank.

Root cause

In particle_mesh_readin.f90 a blocking MPI_ALLGATHERV(MPI_IN_PLACE, …) is called on MPI_COMM_LEADERS_SHARED — the communicator that contains only one rank per compute node (the "leader"). On a single compute node this communicator has exactly one member. MS-MPI's implementation of MPI_IN_PLACE on a 1-process communicator zeroes the entire receive buffer before copying from the in-place source — but because the source is the buffer, the source has already been wiped. The result is that SideInfo_Shared columns 1–9 (including SIDE_LOCALID) are all set to zero. A subsequent call to GetGlobalNonUniqueSideID then finds no side with a matching SIDE_LOCALID and aborts.

Why this only affects single-node runs on Windows On a multi-node cluster MPI_COMM_LEADERS_SHARED has more than one member and the MPI_IN_PLACE gather works correctly. On Linux with a 1-process communicator the data is simply left in place (a no-op). MS-MPI is unique in zeroing the buffer first.

The four MPI_IALLGATHERV(MPI_IN_PLACE, …) calls that precede this one (for ElemInfo_Shared, SideInfo_Shared, NodeInfo_Shared, NodeCoords_Shared on MPI_COMM_SHARED with all 4 procs) are not affected: they run on a 4-process communicator where MPI_IN_PLACE works correctly.

Fix — particle_mesh_readin.f90

Guard the blocking gather with a process-count check so it is skipped when there is only one leader (single compute node):

CALL BARRIER_AND_SYNC(SideInfo_Shared_Win,MPI_COMM_SHARED)
IF (myComputeNodeRank.EQ.0) THEN
  ! Only gather across node leaders when there are multiple compute nodes.
  ! MS-MPI MPI_ALLGATHERV(MPI_IN_PLACE,...) on a 1-process communicator corrupts
  ! the receive buffer by zeroing it before filling from the (now-zero) in-place
  ! buffer. With a single node all data is already in shared memory -- skip.
  IF (nLeaderGroupProcs.GT.1) THEN
    CALL MPI_ALLGATHERV(MPI_IN_PLACE,0,MPI_DATATYPE_NULL,SideInfo_Shared,recvcountSide &
        ,displsSide,MPI_STRUCT_SIDE,MPI_COMM_LEADERS_SHARED,IERROR)
  END IF
  CALL MPI_TYPE_FREE(MPI_STRUCT_ELEM,iError)
  CALL MPI_TYPE_FREE(MPI_STRUCT_SIDE,iError)
  CALL MPI_TYPE_FREE(MPI_STRUCT_NODE,iError)
END IF

File: src/particles/particle_mesh/particle_mesh_readin.f90 (around the MPI_ALLGATHERV … MPI_COMM_LEADERS_SHARED block).

13.2 Intersection-based periodic boundary tracking fails

Symptom

ERROR: Particle not inside of Element, ipart …

Occurs during the first time step; particles near x=0 moving in −x direction are removed from the simulation.

Root cause

The default TrackingMethod = refmapping uses intersection-based tracking at periodic boundaries (ParticleBCTracking). On Windows this routine fails to detect when a particle crosses the periodic x=0 face. After the Boris push, particles near x=0 with negative vx end up at x < 0 (outside the domain). SinglePointToElement correctly returns −1, LocateParticleInElement calls RemoveParticle, and the abort fires.

Root cause is likely a floating-point edge case in the intersection test The Linux build with OpenMPI passes this test. The Windows/MS-MPI build triggers a different code path in the intersection geometry, probably due to slightly different floating-point rounding in MSVC-compatible math routines vs. glibc.

Fix — parameter.ini

Add CartesianPeriodic = T immediately after TrackingMethod: 

TrackingMethod              = refmapping
CartesianPeriodic           = T

This activates PeriodicMovement: instead of computing intersection points at periodic faces, it directly wraps the particle position (if pos > xmax: pos = pos − L, if pos < 0: pos = pos + L). PIClas documents this as "Simplified treatment for periodic box with Refmapping. No computation of intersection points at periodic BCs." It is physically equivalent for a Cartesian periodic domain.

With CartesianPeriodic=T and nPeriodicVectors=3, ParticleBCTracking calls PeriodicMovement which wraps positions in all three directions and immediately cycles/returns, bypassing the problematic intersection calculations.

13.3 Single-node shape_function run: MPI exchange halo too small

Symptom

GlobalProcToExchangeProc(EXCHANGE_PROC_RANK,ProcID) is negative.
The halo region might be too small. Try increasing Particles-HaloEpsVelo!

Abort occurs during the particle MPI exchange at the end of the first time step, on a proc that needs to send a particle to a non-adjacent proc.

Root cause — step by step

GlobalProcToExchangeProc maps global proc IDs to local exchange-proc indices. It is built in particle_mpi_halo.f90:IdentifyPartExchangeProcs using MPI_halo_eps as the distance threshold. A proc can only send particles to procs that appear in its exchange list.

With 4 procs on one node and a domain length of 12.566 (proc width ≈ 3.14), the exchange list for each proc is:

Procx rangeExchange listNOT included
00 – 3.14Proc 1 (right), Proc 3 (left via periodic)Proc 2
13.14 – 6.28Proc 0 (left), Proc 2 (right)Proc 3
26.28 – 9.42Proc 1 (left), Proc 3 (right)Proc 0
39.42 – 12.57Proc 2 (left), Proc 0 (right via periodic)Proc 1

The intended behaviour for single-node runs is: MPI_halo_eps = c × dt × SafetyFactor = 1×108 × 0.1 × 1.2 = 1.2×107 — much larger than the domain, so all four procs are in each other's exchange lists.

The code achieves this through the branch in particle_mpi_halo.f90: 

IF (halo_eps.LE.0.) THEN
  ! single-node path: MPI_halo_eps = c * dt * SafetyFactor  (~1.2e7)
ELSE
  ! multi-node path: MPI_halo_eps = halo_eps                (~1.0)
END IF

For single-node runs without shape_function deposition, particle_bgm.f90 leaves halo_eps = 0, and the correct single-node branch is taken.

For single-node runs with PIC-Deposition-Type = shape_function and PIC-shapefunction-dimension = 1, particle_bgm.f90 sets: 

halo_eps = MAX(r_sf, MAX(y_extent, z_extent))
         = MAX(0.5,  MAX(1.0,     1.0     ))
         = 1.0   > 0

Because halo_eps = 1.0 > 0 the code wrongly enters the multi-node branch and sets MPI_halo_eps = 1.0. With a proc width of 3.14 and halo of only 1.0, procs 1 and 3 (and procs 0 and 2) are not in each other's exchange lists. A particle that acquires a large velocity from a corrupted field interpolation can jump across two proc-widths in one time step and then cannot be sent to the target proc → abort.

Why the field interpolation can be corrupted PEM%LocalElemID(iPart) = PEM%GlobalElemID(iPart) − offsetElem. If a misplaced particle carries a GlobalElemID belonging to a different proc, the resulting LocalElemID exceeds nElems and accesses out-of-bounds memory in U_N(LocalElemID)%E, producing a garbage electric field (~1024), a garbage velocity (~−2.7×1023), and a position that wraps incorrectly (integer overflow in CEILING inside PeriodicMovement).

Fix — particle_mpi_halo.f90

Replace the halo_eps-based test with an explicit single-node test. nComputeNodeProcessors and nProcessors_Global are both available in MOD_MPI_Shared_Vars, which is already USEd without an ONLY clause in this subroutine. 

! OLD (incorrect when shape_function sets halo_eps > 0 on a single node):
IF (halo_eps.LE.0.) THEN

! NEW (correctly detects single-node regardless of halo_eps):
IF (nComputeNodeProcessors.EQ.nProcessors_Global) THEN

File: src/particles/particle_mpi/particle_mpi_halo.f90, subroutine IdentifyPartExchangeProcs, the comment block starting "if running on one node, halo_eps is meaningless".

After this fix, single-node runs with shape_function use the single-node path and obtain MPI_halo_eps ≈ 1.2×107, which covers the entire domain and ensures all procs are in each other's exchange lists.

Rebuild required After applying this fix, rebuild PIClas (see Section 6) and copy the new piclas.exe to the simulation directory before re-running.

14.2 Architecture Overview

The GPU layer is a thin C/CUDA static library (piclasGPU) that is linked into libpiclas.dll and piclas.exe. Fortran calls the CUDA routines via ISO_C_BINDING — no Fortran CUDA extension or special compiler is required on the Fortran side. 

  ┌──────────────────────────────────────────┐
  │  Fortran time-stepping code              │
  │  (compiled by gfortran / MSYS2 UCRT64)   │
  │                                          │
  │  CALL GPU_PushParticlesBatch(...)         │  ← ISO_C_BINDING
  └───────────────┬──────────────────────────┘
                  │ C calling convention
  ┌───────────────▼──────────────────────────┐
  │  gpu_interface.f90  (BIND(C) wrapper)    │
  │  gpu_vars.f90       (state variables)    │
  └───────────────┬──────────────────────────┘
                  │ extern "C"
  ┌───────────────▼──────────────────────────┐
  │  piclasGPU.a  (CUDA static lib)          │
  │   gpu_init.cu      — device init         │
  │   gpu_memory.cu    — buffers + transfer  │
  │   particle_push.cu — CUDA kernel         │
  │  (compiled by nvcc with cl.exe host)     │
  └───────────────┬──────────────────────────┘
                  │ cudaMemcpy + kernel launch
  ┌───────────────▼──────────────────────────┐
  │  NVIDIA GPU (CUDA device)                │
  │   particle_push_kernel<<<grid,block>>>  │
  │   each thread: pos += vel * dt           │
  └──────────────────────────────────────────┘

The CUDA objects are compiled by nvcc using MSVC cl.exe as the host compiler (the only supported host compiler for nvcc on Windows). The resulting COFF objects use the MSVC ABI for host-side code but expose GPU entry points via extern "C", making them linkable from MinGW-compiled Fortran/C code.

14.5 Windows/CUDA Technical Challenges and Solutions

Combining nvcc (MSVC host ABI) with MinGW/gfortran (GNU ABI) on Windows involves a series of non-obvious problems. Each is documented here so future maintainers understand why each workaround exists.

Challenge 1 — COMDAT .bss sections cause ACCESS VIOLATION at DLL load

Symptom: libpiclas.dll fails to load with STATUS_DLL_INIT_FAILED (error 1114) at import time.
Root cause: nvcc (using cl.exe as host) emits two function-local statics as LINK_ONCE_DISCARD COMDAT .bss sections in each .cu.obj: 

?__ref@?1??____nv_dummy_param_ref@@YAXPEAX@Z@4PEAPECXEA
?__ref@?2??__nv_cudaEntityRegisterCallback@@YAXPEAPEAX@Z@4PEAPECXEA

GNU ld discards COMDAT sections, leaving IMAGE_REL_AMD64_REL32 relocations in ?__sti____cudaRegisterAll with displacement zero. This produces mov %rax, 0(%rip) — writing the CUDA fat-binary handle into the read-only .text section → access violation.
Fix: patch_cuda_comdat.py clears IMAGE_SCN_LNK_COMDAT from every .bss section in the COFF object AND zeroes the COMDAT Selection byte in the matching auxiliary symbol record. GNU ld then treats them as regular BSS and always includes them.

Challenge 2 — cudart.lib MSVC static loader causes DLL init ordering crash

Symptom: STATUS_DLL_INIT_FAILED even after COMDAT fix, or intermittent CUDA API failures on first call.
Root cause: NVIDIA's cudart.lib contains cudart_static_loader.obj — an MSVC-compiled .CRT$XCU static initializer that calls LoadLibrary("cudart64_13.dll"). Under GNU ld, .CRT$XCU ordering is not guaranteed; the fat-binary registration constructors in piclasGPU.a may run before cudart64_13.dll is loaded.
Fix: At configure time, SetGPU.cmake runs gendef and dlltool to generate a MinGW-native import library libcudart.dll.a from cudart64_13.dll. This puts cudart64_13.dll in the PE IAT (import address table), so Windows loads it before any DllMain or constructor runs — correct ordering without relying on .CRT$XCU indices. 

# cmake/SetGPU.cmake generates these commands at configure time:
gendef  C:/Program Files/NVIDIA GPU Computing Toolkit/CUDA/v12.x/bin/x64/cudart64_13.dll
dlltool -d cudart64_13.def -l libcudart.dll.a -D cudart64_13.dll

Challenge 3 — nvcc requires MSVC cl.exe (MinGW g++ unsupported)

nvcc on Windows does not support MinGW g++ as a host compiler ("unsupported target OS" error). SetGPU.cmake searches for cl.exe in known VS 2019/2022 install paths and sets CMAKE_CUDA_HOST_COMPILER before ENABLE_LANGUAGE(CUDA). The extern "C" interface in piclas_gpu.h ensures the MSVC-ABI CUDA objects link correctly against the MinGW-compiled Fortran code.

Challenge 4 — CMake CUDA link test fails with MinGW

CMake's ENABLE_LANGUAGE(CUDA) runs a test link using cmake -E vs_link_exe (MSVC protocol) because the host compiler is cl.exe. This drives MinGW's ld with MSVC flags (/nologo, /out:) and windres.exe with MSVC manifest arguments — both fail.
Fix: Pre-set CMAKE_CUDA_COMPILER_WORKS=TRUE and CMAKE_CUDA_ABI_COMPILED=TRUE in the cache before ENABLE_LANGUAGE(CUDA). CMake skips the link test and ABI detection, treating them as already resolved. Since piclasGPU is a static library, no CUDA executable link is ever needed during the build.

Challenge 5 — GNU ar rule overwritten by CUDA language detection

After ENABLE_LANGUAGE(CUDA), CMake sets CMAKE_CUDA_CREATE_STATIC_LIBRARY to an MSVC-style rule ("ar.exe /nologo /out:TARGET OBJECTS"). MinGW ar does not understand /nologo or /out:.
Fix: Immediately after ENABLE_LANGUAGE(CUDA), override: 

SET(CMAKE_CUDA_CREATE_STATIC_LIBRARY
  "<CMAKE_AR> qc <TARGET> <LINK_FLAGS> <OBJECTS>"
  CACHE STRING "GNU ar rule for CUDA static library" FORCE)

Challenge 6 — CUDA implicit link libraries use MSVC bare names

ENABLE_LANGUAGE(CUDA) probes the MSVC toolchain and records all implicit Windows SDK libraries (kernel32, ws2_32, cudart, etc.) as bare names into CMAKE_CUDA_IMPLICIT_LINK_LIBRARIES. MinGW's ld requires -l<name> — bare names are unresolvable.
Fix: Clear both variables immediately after language setup: 

SET(CMAKE_CUDA_IMPLICIT_LINK_LIBRARIES   "" CACHE INTERNAL "")
SET(CMAKE_CUDA_IMPLICIT_LINK_DIRECTORIES "" CACHE INTERNAL "")

Challenge 7 — CUDA language detection clobbers global linker flags

Including Platform/Windows-NVIDIA-CUDA.cmake pulls in Platform/Windows-MSVC.cmake, which sets CMAKE_LINK_LIBRARY_FLAG="" (empty, MSVC style) and CMAKE_LIBRARY_PATH_FLAG="-LIBPATH:" globally. This strips the -l prefix from all subsequent gfortran/ld link commands — HDF5, MPI, and PETSc all stop linking.
Fix: 

SET(CMAKE_LINK_LIBRARY_FLAG "-l")
SET(CMAKE_LIBRARY_PATH_FLAG "-L")
SET(CMAKE_CUDA_LINKER_PREFERENCE_PROPAGATES 0)

Challenge 8 — .voltbl section makes libpiclas.dll malformed

MSVC-compiled CUDA objects contain a .voltbl section (volatile-metadata / Control-Flow-Guard table). GNU ld assigns it an address outside the normal DLL address range, producing STATUS_INVALID_IMAGE_FORMAT — Windows refuses to load the DLL.
Fix: Strip the section post-link with objcopy: 

C:/msys64/ucrt64/bin/objcopy.exe --remove-section=.voltbl libpiclas.dll

This is an automatic CMake POST_BUILD step on the libpiclasshared target.

Challenge 9 — MSVC runtime symbols undefined in MinGW link

CUDA objects compiled with cl.exe (and NVIDIA's pre-compiled cudart.lib) reference MSVC-specific symbols that do not exist in MinGW's libraries: __security_cookie, __security_check_cookie, __GSHandlerCheck, __local_stdio_printf_options, _Init_thread_epoch, _Init_thread_header, _Init_thread_footer.
Fix: src/gpu/msvc_stubs.c compiled by MinGW gcc provides inert no-op stubs for all of these. It is built into piclasGPUStubs.a and linked with --whole-archive to guarantee all stubs are present.

Challenge 10 — api-ms-win-crt-private dependency appears in DLL import table

MSVC-compiled objects in cudart.lib (compiled with /MD) reference basic CRT functions (memcpy, memcmp, strchr, etc.) via MSVC DLL import thunks: call [__imp_memcpy]. GNU ld creates IAT entries for api-ms-win-crt-private-l1-1-0.dll, which is in System32\downlevel\ (not on the normal DLL search path) → runtime DLL-not-found failure.
Fix: msvc_stubs.c defines __imp_memcpy, __imp_memcmp, etc. as function pointers to MinGW's own implementations. GNU ld resolves the MSVC relocations to these local pointer variables — no IAT entry is created and the private UCRT DLL dependency disappears entirely.

Challenge 11 — Thread-safe static init stubs with <<<>>> syntax

CUDA <<<>>> kernel launch syntax causes nvcc to generate a host-side stub containing a function-local static with MSVC thread-safe initialization (_Init_thread_header / _Init_thread_footer / _Init_thread_epoch). Even with /Zc:threadSafeInit-, some stubs remain.
Fix: particle_push.cu uses the explicit cudaLaunchKernel() API instead of <<<>>>: 

void *args[] = { &d_PartState, &d_isActive, &nPart, &dt };
cudaLaunchKernel((const void*)particle_push_kernel, grid, block, args, 0, nullptr);

Challenge 12 — CMAKE_CUDA_CREATE_SHARED_LIBRARY override silently ignored (31 LNK2019 on __imp_fminf / C99 float math)

Symptom: bin\libpiclasGPU.dll fails to link with 31 LNK2019 errors on __imp_fminf, __imp_fmaxf, __imp_exp2f, __imp_cbrtf, __imp_erff, __imp__dsign, and 25 more C99 float/double math symbols. The linker warning LNK4098: MSVCRT conflicts with other libraries also appears. The build log shows nvcc -shared -cudart shared ... — but the CMAKE_CUDA_CREATE_SHARED_LIBRARY override in src/CMakeLists.txt specifies -cudart none. The override is not being applied.

Root cause (two layers):

  1. CMake's CUDA platform module (loaded during ENABLE_LANGUAGE(CUDA) in cmake/SetGPU.cmake) stores CMAKE_CUDA_CREATE_SHARED_LIBRARY as a CACHE INTERNAL variable — with FORCE. A subsequent plain SET(CMAKE_CUDA_CREATE_SHARED_LIBRARY ...) without FORCE cannot override a cached value and is silently ignored, even in a fresh configure. The platform default (-cudart shared with no /DEFAULTLIB flags) survives into the generated Ninja rule.
  2. With -cudart shared, nvcc auto-links cudadevrt.lib, which injects /DEFAULTLIB:LIBCMT (static MSVC CRT). The MSVC linker then resolves UCRT symbols through libucrt.lib (static UCRT) rather than ucrt.lib (dynamic UCRT import library). libucrt.lib does not expose __imp_fminf / __imp_fmaxf / … DLL-import stubs — those live only in ucrt.lib — so every C99 float math overload that CUDA's device wrapper headers generate becomes an unresolved external.

Fix: Add CACHE INTERNAL "" FORCE to the SET(CMAKE_CUDA_CREATE_SHARED_LIBRARY ...) call in src/CMakeLists.txt so it runs after the platform module's forced SET and wins: 

SET(CMAKE_CUDA_CREATE_SHARED_LIBRARY
    "<CMAKE_CUDA_COMPILER> -shared -cudart none -ccbin \"...\" -o <TARGET> <OBJECTS>
     -Xlinker /LIBPATH:\"...\" -Xlinker cudart.lib
     -Xlinker /NODEFAULTLIB:LIBCMT -Xlinker /NODEFAULTLIB:LIBCMTD
     -Xlinker /NODEFAULTLIB:libucrt -Xlinker /DEFAULTLIB:ucrt"
    CACHE INTERNAL "CUDA SHARED library link rule (Windows MinGW+MSVC host)" FORCE)

With -cudart none the automatic cudadevrt/cudart library injection is suppressed; cudart.lib is added explicitly instead. /NODEFAULTLIB:LIBCMT and /NODEFAULTLIB:libucrt remove the static-CRT defaults; /DEFAULTLIB:ucrt adds the dynamic UCRT import library which provides all 31 missing __imp_* stubs. After applying this fix a fresh cmake --preset windows-ucrt64-gpu (reconfigure) is required so CMake regenerates build.ninja with the corrected link rule.

Secondary failure after applying the CACHE FORCE fix — LNK1181 on "Files\NVIDIA.obj"
Once the template is applied, a second error can appear: LINK : fatal error LNK1181: cannot open input file 'Files\NVIDIA.obj'. This is caused by the /LIBPATH: argument containing spaces (C:/Program Files/NVIDIA GPU Computing Toolkit/…). When Ninja wraps the link rule in cmd.exe /C "…", the inner quotes around /LIBPATH:"…" are consumed by cmd.exe's quote parser. nvcc receives the argument with its quotes stripped and passes it to link.exe without re-quoting; link.exe then splits the path at the first space and treats Files\NVIDIA as a stray object file.

Fix (already in src/CMakeLists.txt): at configure time, copy cudart.lib from the CUDA toolkit into the build directory (${CMAKE_BINARY_DIR}/lib/cuda/), which is assumed to be space-free (e.g. C:/Data/PRJ/piclas-win/dsmc-cone-2D/lib/cuda). Then use that path in /LIBPATH: — no quotes needed: 
SET(_CUDA_LOCAL_LIB_DIR "${CMAKE_BINARY_DIR}/lib/cuda")
FILE(MAKE_DIRECTORY "${_CUDA_LOCAL_LIB_DIR}")
FILE(COPY "${CUDAToolkit_LIBRARY_DIR}/cudart.lib" DESTINATION "${_CUDA_LOCAL_LIB_DIR}")
# Template uses space-free build-dir path — no quoting required
-Xlinker /LIBPATH:${_CUDA_LOCAL_LIB_DIR} -Xlinker cudart.lib
CMake's FILE(COPY ...) runs at configure time, so cudart.lib is present in the build tree before Ninja starts. During configure you should see: -- CUDA cudart.lib staged: .../dsmc-cone-2D/lib/cuda/cudart.lib
Note — PowerShell GetShortPathName() approach was tried first
An earlier attempt used [System.IO.Path]::GetShortPathName() to obtain the 8.3 form of the path. This failed because PowerShell may not be in the cmake subprocess's PATH, or 8.3 names may be disabled on the volume. The copy-to-build-dir approach is unconditionally reliable as long as the build directory itself has no spaces.

14.8 GPU Init / Finalize Call Sites

GPU_Init and GPU_Finalize are called inside the particle initialization and finalization routines using Fortran's BLOCK construct, which scopes the USE statements to avoid polluting the module's namespace.

Initialization — src/particles/particle_init.f90, InitParticles()

Called at the end of InitParticles, after PDM%maxParticleNumber has been set and all particle arrays (PartState, PDM%ParticleInside, etc.) have been allocated:

#if PICLAS_USE_GPU
! Initialize GPU: locate device, allocate device buffers for all particles
BLOCK
  USE MOD_GPU_Interface, ONLY: GPU_Init
  USE MOD_Particle_Vars, ONLY: PDM
  CALL GPU_Init(PDM%maxParticleNumber)
END BLOCK
#endif /*PICLAS_USE_GPU*/

ParticlesInitIsDone=.TRUE.

GPU_Init calls piclas_gpu_init() (prints device info), piclas_gpu_alloc_buffers(nMaxPart) (allocates device-side PartState + isActive arrays), and GPU_AllocActiveMask(nMaxPart) (allocates the host-side integer mask buffer). After this call, GPUInitialized = .TRUE..

Finalization — src/particles/particle_init.f90, FinalizeParticles()

Called at the end of FinalizeParticles, before the subroutine returns. The GPUInitialized guard prevents double-free if GPU was never initialized (e.g., non-GPU builds that somehow reach this path):

#if PICLAS_USE_GPU
! Finalize GPU: free device buffers and reset CUDA device
BLOCK
  USE MOD_GPU_Interface, ONLY: GPU_Finalize
  USE MOD_GPU_Vars,      ONLY: GPUInitialized
  IF (GPUInitialized) CALL GPU_Finalize()
END BLOCK
#endif /*PICLAS_USE_GPU*/

GPU_Finalize frees the host mask buffer, calls piclas_gpu_free_buffers() (frees device arrays), then piclas_gpu_finalize() (cudaDeviceReset()). Sets GPUInitialized = .FALSE..

14.9 DSMC Time-Step Integration

The GPU batch push is integrated into src/timedisc/timedisc_TimeStep_DSMC.f90 (PP_TimeDiscMethod==4) using three coordinated changes.

Step A — Module imports (top of subroutine)

#if PICLAS_USE_GPU
USE MOD_GPU_Vars      ,ONLY: GPUInitialized
USE MOD_GPU_Interface ,ONLY: GPU_PushParticlesBatch
#endif /*PICLAS_USE_GPU*/

And the local flag variable:

#if PICLAS_USE_GPU
LOGICAL :: UseGPUPush
#endif /*PICLAS_USE_GPU*/

Step B — Eligibility check before the particle loop

The GPU push is only valid for the simple constant-dt case. It is disabled when any of the following features are active, because they either require per-particle dtVar or modify positions inside the loop in ways that the GPU kernel does not replicate: 

#if PICLAS_USE_GPU
UseGPUPush = GPUInitialized          .AND. &
             .NOT.UseVarTimeStep     .AND. &  ! GPU kernel uses constant dt
             .NOT.UseRotRefFrame     .AND. &  ! RotRef uses CalcPartPosInRotRef
             .NOT.UseGranularSpecies .AND. &  ! granular particles have their own push
             .NOT.DoSurfaceFlux      .AND. &  ! surface-flux particles use dtFracPush (random dt)
             .NOT.Symmetry%Axisymmetric        ! axisym calls CalcPartSymmetryPos on new positions
#endif /*PICLAS_USE_GPU*/

Step C — CPU loop: skip position update for GPU-eligible particles

The per-particle CPU loop still runs (it handles LastPartPos bookkeeping, surface-flux fractional-dt particles, and rotating-reference-frame particles). For GPU-eligible particles the position update is skipped with CONTINUE — the GPU kernel will do it below the loop: 

    IF(UseRotRefFrame) THEN
      LastPartVeloRotRef(1:3,iPart)=PartVeloRotRef(1:3,iPart)
      CALL CalcPartPosInRotRef(iPart, dtVar)
#if PICLAS_USE_GPU
    ELSE IF (UseGPUPush) THEN
      CONTINUE   ! position push offloaded to GPU batch call below the loop
#endif /*PICLAS_USE_GPU*/
    ELSE
      PartState(1:3,iPart) = PartState(1:3,iPart) + PartState(4:6,iPart) * dtVar
    END IF
LastPartPos is still set correctly LastPartPos(1:3,iPart) = PartState(1:3,iPart) executes inside the IF (.NOT.PDM%dtFracPush(iPart)) branch, which runs before the RotRef/GPU/CPU fork. So LastPartPos captures the position at the start of the time step regardless of which path does the actual push.

Step D — GPU batch call after the loop

#if PICLAS_USE_GPU
! GPU batch position push: upload PartState, run kernel, download results.
! Replaces the per-particle CPU push skipped with CONTINUE inside the loop.
IF (UseGPUPush) THEN
  CALL GPU_PushParticlesBatch(PartState, PDM%ParticleInside, PDM%ParticleVecLength, dt)
END IF
#endif /*PICLAS_USE_GPU*/

GPU_PushParticlesBatch converts PDM%ParticleInside (Fortran LOGICAL) to a C int mask in GPU_ActiveMask, then calls piclas_gpu_push_particles:

  1. H2D cudaMemcpy — uploads PartState(1:6, 1:nPart) and the mask to device
  2. Kernel launch — particle_push_kernel<<<ceil(nPart/256), 256>>> sets pos += vel * dt for every active particle (one thread each)
  3. D2H cudaMemcpy — downloads updated PartState back to host (implicit synchronisation point — no extra cudaDeviceSynchronize needed)
Memory layout Fortran stores PartState(1:6, iPart) column-major: all 6 components of particle iPart are contiguous in memory. The CUDA kernel maps iPart * 6 + 0..2 → x,y,z and iPart * 6 + 3..5 → vx,vy,vz, matching this layout exactly. No transposition is needed.

14.10 LSERK4 Time-Step Integration (PIC-EM, Step 2) NEW

Extends GPU acceleration to PP_TimeDiscMethod 2 (LSERK4-5, Carpenter 1994, 5 stages) and PP_TimeDiscMethod 6 (LSERK4-14, Niegemann 2012, 14 stages) — the two methods used for PIC-EM and combined DSMC+PIC-EM simulations. Both methods call the same TimeStepByLSERK() subroutine; only the compile-time RK coefficients and stage count differ.

Design guideline — Pt_temp residency (from Linux MPI source analysis)

Pt_temp follows the Linux MPI design: persistent host array, communicated with migrating particles

In the Linux MPI code (particle_mpi.f90), Pt_temp(1:6, iPart) is a persistent host-side array that lives for the entire timestep across all RK stages. When a particle crosses a domain boundary mid-sweep, the MPI layer packs Pt_temp(1:6) into the send buffer and unpacks it on the receiving rank — the accumulated derivative history travels with the particle so the RK integration can continue correctly without recomputation.

The GPU implementation mirrors this design exactly:

  • d_Pt_temp is allocated alongside d_PartState and initialized to zero (matching Pt_temp=0. in particle_init.f90).
  • Per stage: PartState and Pt_temp are uploaded (H2D) before the kernel and downloaded (D2H) after. The D2H of Pt_temp keeps the host copy in sync, which is the correct foundation for future MPI+GPU support — when a particle crosses a rank boundary after the kernel, the host Pt_temp already contains the updated value.
  • Pt (acceleration from Lorentz force), isActive, isPush, and isNewPart are also uploaded each stage.
  • For single-node builds (current target) there is no MPI particle exchange, so the Pt_temp D2H is cheap insurance rather than a correctness requirement.

15. superB Windows Port NEW

15.1 What superB Does

superB is a standalone magnetic-field pre-processor bundled with PICLas. It is a separate executable (not part of the main simulation loop) that:

  1. Reads a PICLas mesh file (HDF5).
  2. Evaluates coil magnetic fields via the Biot-Savart law (static) or Jefimenko equations (time-dependent), and permanent-magnet fields via magnetic scalar potential + gradient.
  3. Writes the resulting background B-field (BGField) to an HDF5 output file.
  4. That HDF5 file is referenced in the main PICLas parameter file so the field is loaded at runtime as a frozen background field.

Typical use case: compute the static field of a Helmholtz coil or a permanent-magnet ring, then run a combined DSMC+PIC-EM simulation that accelerates charged particles in that field.

15.2 Build Architecture

When PICLAS_PARTICLES=ON (the default for all maxwell/DSMC builds), CMake includes all src/posti/superB/*.f90 files in the main particlesF90 glob that feeds libpiclasshared. The physics modules (superB_coil, superB_permMag, superB_init, superB_main, superB_tools, superB_vars) are therefore compiled into libpiclas.dll as a side effect of any full PICLas build.

When PICLAS_BUILD_POSTI=ON the posti CMakeLists adds the superB standalone target: 

superB.exe ──links──► superBlib   (OBJECT lib: superB.f90 entry point)
                    ► libpiclasshared  (mesh, HDF5, all superB physics)
                    ► ${linkedlibs}    (HDF5, OpenBLAS — WIN32 add_exec branch)
No extra DLLs needed for superB. Because the physics is already in libpiclas.dll, the superB.exe binary is tiny (~100 KB). It shares the same HDF5 and MPI infrastructure as piclas.exe and does not require any GPU libraries.

15.3 Windows Changes

The superB Fortran source is entirely cross-platform — no POSIX calls, no signal handlers, no Linux-specific I/O. Only two build-system changes were required:

Change 1 — New windows-ucrt64-superB CMake preset (CMakePresets.json)

The base windows-ucrt64 preset has PICLAS_BUILD_POSTI=OFF. A dedicated preset activates the superB target without requiring the user to pass extra -D flags: 

{
  "name": "windows-ucrt64-superB",
  "inherits": "windows-ucrt64",
  "binaryDir": "${sourceDir}/build-ucrt64-superB",
  "cacheVariables": {
    "PICLAS_EQNSYSNAME":     "maxwell",
    "PICLAS_TIMEDISCMETHOD": "RK4",
    "PICLAS_BUILD_POSTI":    "ON",
    "PICLAS_UNITTESTS":      "OFF"
  }
}

maxwell + RK4 is chosen because superB output is typically used as a background field in PIC-EM simulations. Any equation system works — superB itself only uses the mesh and HDF5 infrastructure.

The Build Configurator (piclas_builder.html) already has a PICLAS_BUILD_POSTI checkbox. Its description and build target line were updated to mention superB, so ticking the box now builds piclas piclas2vtk superB instead of just piclas piclas2vtk.

Change 2 — Regex case fix in src/posti/superB/CMakeLists.txt

The PICLAS_PARTICLES=OFF branch used a lowercase regex to exclude the standalone entry point from the library source list: 

# Before (broken on case-sensitive and case-preserving filesystems):
LIST(FILTER superBnoparticlesF90 EXCLUDE REGEX ".*superb.f90$")

# After:
LIST(FILTER superBnoparticlesF90 EXCLUDE REGEX ".*[Ss]uper[Bb]\\.f90$")

The actual filename is superB.f90 (capital B). CMake REGEX is case-sensitive, so the original pattern silently failed to exclude the PROGRAM entry point from the library object list, which would cause a duplicate-symbol link error on Windows in no-particles builds. The fix uses a character class to match either case.

Scope: This code path is only active when PICLAS_PARTICLES=OFF (uncommon — requires explicitly disabling particle physics). All standard maxwell/DSMC/RK4 builds use PICLAS_PARTICLES=ON (the default) and are not affected.

15.4 Building & Running

Build

Use the dedicated preset to configure and build:

cmake --preset windows-ucrt64-superB -B build-ucrt64-superB
cmake --build build-ucrt64-superB --target superB

Or check PICLAS_BUILD_POSTI in the Build Configurator and select any equation system — the downloaded .bat will build piclas piclas2vtk superB together.

Run

cd my-superB-case
..\build-ucrt64-superB\bin\superB.exe superB.ini

superB.ini is the parameter file. It must specify the mesh file (MeshFile = ...) and the coil/magnet geometry. On success superB writes *_BGField.h5 (and optionally *_BGField_TDep.h5 for time-dependent fields).
